# si-project-docs — SI 프로젝트 산출물 원천 문서 스킬

## 핵심 정체성

이 스킬은 한국 SI 프로젝트에서 클라이언트·감리·PM에게 제출되는 **정식 산출물의 변환 원천(source of truth)** 을 markdown으로 관리한다.

- **무엇을 만드는가**: SI 프로젝트 표준 산출물 8단계(착수기획~관리) × 40여 종에 해당하는 markdown 문서
- **누가 쓰는가**: PM·PL·SE·QA 등 프로젝트 관계자가 이 markdown을 보고 검토하거나, 별도 변환 스킬을 통해 docx/xlsx/HTML/PDF로 변환하여 최종 산출물로 제출
- **왜 markdown인가**: 단일 원천(source of truth)을 유지하면, 형식(엑셀·워드·HTML)이 달라도 내용·구조가 일관되게 유지된다. 형식 변환은 별도 스킬이 담당한다.

> **이 스킬은 형식이 아니라 내용·구조의 정확성에 집중한다.** 표 컬럼, ID 체계, 항목 누락, 메타데이터 일관성, 도메인 분할 규칙 등이 우선이며, 출력 포맷(폰트, 셀 병합, 머리말/꼬리말 등)은 변환 단계에서 다뤄진다.

## 산출물 관리 원칙

이 스킬은 **산출물 유형별 단일 파일** 관리를 기본으로 한다.

- **기본 동작**: 기존 파일이 있으면 해당 파일을 **업데이트** (새 섹션 추가, 기존 내용 보완)
- **분리 요청 시**: 사용자가 명시적으로 도메인별 분리를 요청하면 `{문서명}_{도메인}.md` 형식으로 파일을 나눈다 (예: `API명세서_USER.md`, `API명세서_ORDER.md`)

## 출력 디렉토리 규칙

산출물은 SI 표준 8단계 구조를 그대로 반영하여 `docs/{단계}/{문서명}.md` 로 생성한다.

```
docs/
├── 01_착수기획/
│   ├── 프로젝트헌장.md
│   └── 사업기획서.md
├── 02_분석/
│   ├── 요구사항정의서.md
│   └── 현행시스템분석서.md
├── 03_설계/
│   ├── 시스템아키텍처설계서.md
│   ├── API명세서.md
│   └── 화면설계서.md
├── 04_구축/
├── 05_시험/
├── 06_전개/
├── 07_운영/
└── 08_관리/
```

- `references/` 폴더의 단계 구조(`01_착수기획/`, `02_분석/`, …)를 그대로 따른다.
- 사용자가 특정 경로를 명시하면 그 경로를 우선한다.
- 기존 프로젝트가 다른 디렉토리 컨벤션(예: 평탄형 `docs/`)을 이미 쓰고 있다면 그 컨벤션을 유지한다.

## R&R 협의 가이드

산출물 작성 중 불확실한 요소가 발생하면 담당자와 협의가 필요하다. 사용자에게 적절한 협의 대상을 안내한다.

| 구분              | 협의 대상 | 내용                                  |
| ----------------- | --------- | ------------------------------------- |
| **시스템 구성**   | SE 담당자 | 인프라, 서버 구조, 네트워크 등        |
| **기술 스펙**     | PL 담당자 | 라이브러리 버전, 아키텍처, 컨벤션 등  |
| **요구사항 분석** | PM 담당자 | 비즈니스 로직, 기능 범위, 우선순위 등 |

문서 내용 중 위 영역에 해당하는 불확실한 항목이 있으면, 플레이스홀더로 남기고 "이 항목은 {SE/PL/PM} 담당자와 협의가 필요합니다"라고 안내한다. 정식 산출물 변환 전에 모든 플레이스홀더가 채워져야 한다.

## 원천 데이터(source data) 요구사항

이 스킬은 산출물의 **변환 원천**을 만드는 스킬이므로, 그 자체로 또 다른 **입력 원천(source data)** 이 필요하다. 산출물마다 필요한 입력 종류가 다르다.

### 산출물별 주요 원천 데이터

| 산출물                | 필요한 원천 데이터 (예시)                                                                          |
| --------------------- | -------------------------------------------------------------------------------------------------- |
| 요구사항정의서        | 기획서, 이해관계자 인터뷰 메모, RFP, 현행 시스템 분석 자료                                          |
| 현행시스템분석서      | 기존 시스템 코드/문서, 운영 환경 정보, 기존 ERD/API 명세                                            |
| 시스템구성도          | 인프라 정보(서버·네트워크·존), SE가 제공한 구성 자료, 클라우드 콘솔 캡처                            |
| 보안정책정의서        | 사내 보안 가이드, 컴플라이언스 요건(ISMS 등), 클라이언트 보안 요구사항                              |
| 시스템아키텍처설계서  | 기술 스택, 인프라 구성, 모듈/서비스 분할안, 외부 연동 시스템 목록                                   |
| 화면설계서            | 와이어프레임/시안, 라우터·페이지 구조, 프론트엔드 소스 또는 기획서                                  |
| 데이터베이스설계서    | DB DDL, JPA Entity 코드, 기존 ERD, 도메인 모델 설명                                                 |
| API명세서             | 컨트롤러 소스, OpenAPI/Swagger 스펙, Postman/Insomnia 컬렉션, RestDocs 결과물                       |
| 인터페이스설계서      | 외부 시스템 API 명세, 연계 시스템 사양서, 메시지 포맷(JSON/XML/EDI) 정의                            |
| 프로그램사양서        | 대상 소스 코드, 메소드/함수 시그니처, 비즈니스 로직 설명                                            |
| 시퀀스다이어그램      | 호출 흐름이 보이는 소스 코드, 비즈니스 시나리오, 컨트롤러→서비스→리포지토리 경로                    |
| 이행계획서            | 이행 대상 데이터 목록, 마이그레이션 스크립트, 컷오버 시나리오                                       |
| ERD(물리모델)         | 실제 DB 스키마, 인덱스/제약 조건, JPA Entity                                                        |
| 테스트케이스          | 요구사항정의서(`REQ-*` ID), 기능 명세, 화면설계서                                                   |
| 단위/통합테스트결과서 | 테스트 실행 로그, 커버리지 리포트, CI 빌드 결과                                                     |
| 성능테스트보고서      | 부하 시나리오, 측정 도구 결과(jMeter/k6/nGrinder), 모니터링 지표                                    |
| 보안취약점점검결과서  | 정적분석/동적분석 도구 리포트, 모의해킹 결과                                                        |
| 배포계획서            | 배포 환경 정보(스테이지/프로덕션), CI/CD 파이프라인, 롤백 절차                                      |
| 릴리즈노트            | git log/태그, 변경 이슈 목록(Jira/Redmine), PR 머지 이력                                            |
| 백업및복구계획서      | 인프라/DB 구성, RTO/RPO 정책, 백업 도구 설정                                                        |

위 표는 모든 경우를 망라하지 않는다. 새로운 산출물이거나 reference에 추가 항목이 정의돼 있으면 해당 reference를 먼저 읽고 필요한 원천을 도출한다.

### 원천이 부족할 때의 동작 (필수)

원천 데이터가 부족하거나 모호한 채로 **임의 추정으로 산출물을 채우지 않는다.** 다음 순서대로 처리한다.

1. **명시적 차단**: 작성을 중단하고, 사용자에게 어떤 원천이 더 필요한지 알린다.
2. **요청 양식**: 다음 항목을 모두 포함해 한 번에 정리해서 요청한다.
   - 작성 대상 산출물명
   - 현재 보유한 원천 (있다면)
   - 부족한 원천의 종류와 형식 (예: "DDL 스크립트(.sql) 또는 JPA Entity 디렉터리 경로")
   - 그 원천이 어디로 매핑되는지 (예: "DB DDL → 데이터베이스설계서의 §3 테이블 정의에 매핑")
   - R&R상 누구에게 받아야 하는지 (SE/PL/PM)
3. **부분 진행 옵션**: 부족한 부분이 일부일 때, 사용자가 "그 부분은 `(TBD)` 마커로 두고 나머지는 작성해줘"라고 명시적으로 허용한 경우에만 부분 진행한다. 사용자 허락 없이 임의로 `(TBD)`를 양산하지 않는다.

**요청 메시지 형식 예시:**

> 데이터베이스설계서를 작성하려면 다음 원천이 필요합니다.
>
> | 항목         | 필요 형식                           | 매핑 위치              | 협의 대상 |
> | ------------ | ----------------------------------- | ---------------------- | --------- |
> | DB 스키마    | DDL `.sql` 또는 `SHOW CREATE TABLE` | §3 테이블 정의         | PL        |
> | JPA Entity   | 엔티티 클래스 디렉터리 경로         | §3 컬럼/제약, §4 ERD   | PL        |
> | 도메인 설명  | 도메인별 책임 메모(또는 구두 설명)  | §1 개요, §2 ERD 그룹핑 | PM        |
>
> 위 중 어떤 것을 어떤 형태로 제공해주실 수 있나요?

## 일괄 요청 처리 정책

사용자가 **여러 산출물을 한 번에**(또는 "전부 다 만들어줘"처럼 광범위하게) 요청한 경우, 바로 실행하지 않는다.

### 일괄 요청 감지 트리거

- "착수기획 산출물 다 만들어줘"
- "설계 단계 전부 작성해줘"
- "{프로젝트}의 산출물 다 채워줘"
- 한 번에 3종 이상 산출물명을 나열한 요청
- "이 프로젝트 산출물 일괄로 / 한꺼번에 / 통째로 만들어줘"

### 일괄 요청 처리 순서 (필수)

1. **실행 보류**: 어떤 파일도 생성·수정하지 않는다.
2. **작성 계획 정리**: 다음 항목을 표로 정리해 사용자에게 보여준다.

   | # | 단계 | 산출물 | 출력 경로 | 필요한 원천 (보유/부족) | 예상 분량 |

3. **원천 부족 항목 별도 표시**: 위 표에서 "부족"으로 표시된 항목은 [원천이 부족할 때의 동작] 절차에 따라 한 번에 모아서 사용자에게 요청한다.
4. **승인 대기**: 사용자가 다음 중 하나로 응답할 때까지 작성을 시작하지 않는다.
   - 계획 그대로 진행 → 모든 산출물 작성 시작
   - 우선순위/순서 변경 → 수정된 계획 재제시
   - 일부만 진행 → 해당 항목만 작성
   - 원천 추가 제공 → 보유 현황 갱신 후 재제시
5. **승인 후 실행**: 승인된 항목만, 승인된 순서로 작성한다. 작성 중간에 추가 원천이 필요해지면 다시 보류하고 [원천 부족 시 동작]으로 돌아간다.

### 단건 요청과의 차이

단건 요청(산출물 1개)일 때는 일괄 계획 단계를 건너뛰고, 원천 확인 → 작성으로 바로 진행한다. 다만 단건이라도 원천이 부족하면 동일하게 보류하고 사용자에게 요청한다.

## 워크플로우

### Step 1: 요청 유형 판별 (단건 vs 일괄)

사용자 요청에 포함된 산출물 수와 표현을 보고 단건/일괄을 판별한다.

- **단건**: 산출물 1종 명확히 지정 → Step 2로 진행
- **일괄**: 산출물 3종 이상, "전부", "일괄", "다 만들어줘" 등의 표현 → [일괄 요청 처리 정책]을 따라 계획을 먼저 제시하고 승인받은 뒤 Step 2부터 항목별로 반복

### Step 2: 프로젝트 규모 판단

사용자의 프로젝트 상황을 파악하여 적절한 가이드를 선택한다.

| 규모   | 팀 규모 | 기간     | 필수 산출물 | 가이드                        |
| ------ | ------- | -------- | ----------- | ----------------------------- |
| 소규모 | 2~10명  | 1~3개월  | 9종         | `references/소규모_가이드.md` |
| 중규모 | 10~30명 | 3~12개월 | 15종        | `references/중규모_가이드.md` |
| 대규모 | 30명+   | 12개월+  | 40종+       | `references/대규모_가이드.md` |

프로젝트 규모가 이전에 결정되었거나 `docs/` 디렉터리에 기존 문서가 있으면 해당 규모를 유지한다.
규모를 모르면 사용자에게 한 번만 확인한다.

### Step 3: 대상 문서 결정

사용자 요청에서 작성/업데이트할 문서를 식별한다.

**문서 매핑표:**

| 단계     | 산출물                | reference 경로                                 |
| -------- | --------------------- | ---------------------------------------------- |
| 착수기획 | 사업기획서            | `references/01_착수기획/사업기획서.md`         |
| 착수기획 | 프로젝트헌장          | `references/01_착수기획/프로젝트헌장.md`       |
| 착수기획 | 타당성분석서          | `references/01_착수기획/타당성분석서.md`       |
| 착수기획 | SWOT분석서            | `references/01_착수기획/SWOT분석서.md`         |

### Step 4: reference 파일 읽기

대상 문서의 reference 파일을 Read 도구로 읽어서 작성 방법, 필수 섹션, 표 구조를 확인한다.
규모별 가이드에서 해당 문서의 Tier(필수/선택/생략)도 확인한다.

### Step 5: 원천 데이터(source data) 확인

[원천 데이터(source data) 요구사항] 절을 적용한다.

1. 산출물별 주요 원천 데이터 표와 reference에서 도출한 필수 항목을 기준으로, 현재 보유한 원천을 점검한다.
2. **부족하면 작성을 보류**하고 [원천이 부족할 때의 동작] 절차에 따라 사용자에게 표 형식으로 일괄 요청한다.
3. 보유 원천만으로 충분하거나 사용자가 부분 진행을 명시적으로 허용한 경우에만 Step 6로 진행한다.

### Step 6: 기존 문서 확인 및 작성/업데이트

1. **기존 파일 탐색**: `docs/{단계}/` 디렉터리에서 해당 문서명으로 Glob 검색 (단계 경로를 모르면 `docs/**/{문서명}.md` 패턴 사용)
2. **파일이 있으면**: 기존 내용을 Read로 읽고, 요청된 부분만 Edit으로 업데이트
3. **파일이 없으면**: `docs/{단계}/{문서명}.md` 경로에 reference 구조를 따라 새 파일을 Write로 생성
   - 단계 디렉터리(`docs/03_설계/` 등)가 없으면 함께 생성

### Step 7: 버전 이력 업데이트

문서 하단의 변경 이력 테이블에 변경 내용을 추가한다. 이 이력은 변환된 정식 산출물에도 그대로 옮겨지므로 정확히 관리한다.

```markdown
## 변경 이력

| 버전 | 일자       | 작성자 | 변경 내용            |
| ---- | ---------- | ------ | -------------------- |
| v1.0 | 2026-04-01 | 홍길동 | 초안 작성            |
| v1.1 | 2026-04-02 | 홍길동 | 주문 도메인 API 추가 |
```

## 도메인별 파일 분리 (선택적)

사용자가 **명시적으로** 도메인별 분리를 요청한 경우에만 적용한다.

**트리거 문구 예시:**

- "API명세서를 도메인별로 나눠줘"
- "USER랑 ORDER 설계서 분리해줘"
- "화면설계서를 기능별로 쪼개줘"

**분리 규칙:**

- 파일명: `{문서명}_{도메인}.md` (예: `API명세서_USER.md`, `화면설계서_ORDER.md`)
- 같은 단계 디렉터리 안에 배치 (예: `docs/03_설계/API명세서_USER.md`)
- 각 파일은 동일한 메타데이터 헤더를 가지되, 범위(scope)를 명시
- 목록 파일(`{문서명}_INDEX.md`)을 생성하여 분리된 파일들을 참조

**분리하지 않는 경우 (기본):**

- 하나의 파일 안에서 도메인별 섹션(`## 2.1 사용자 관리 API`, `## 2.2 주문 관리 API`)으로 구분

## 메타데이터 헤더 표준

모든 산출물은 다음 메타데이터 헤더로 시작한다. 이 헤더의 항목은 변환된 정식 산출물(워드/엑셀 등)의 표지 또는 머리말로 그대로 매핑된다.

```markdown
# {문서 제목}

| 항목       | 내용          |
| ---------- | ------------- |
| 문서명     | {문서명}      |
| 프로젝트명 | {프로젝트명}  |
| 버전       | v1.0          |
| 작성일     | {YYYY-MM-DD}  |
| 작성자     | {이름 (역할)} |
| 승인자     | {이름 (역할)} |
```

프로젝트명, 작성자 등은 기존 문서나 프로젝트 컨텍스트에서 가져온다.
정보가 없으면 `{프로젝트명}` 등 플레이스홀더로 남기고 사용자에게 알린다.

## 작성 품질 기준 (변환 원천으로서의 요구사항)

이 markdown은 추후 다양한 포맷으로 변환되므로, 다음 기준을 준수한다.

- **표 중심**: 핵심 정보는 마크다운 표로 정리한다. 변환 단계에서 표 구조가 그대로 엑셀/워드 표로 매핑된다.
- **ID 체계 유지**: 기존 문서의 ID 넘버링 규칙을 따른다 (예: `REQ-FUNC-001`, `API-AUTH-001`, `SCR-DASH-001`). RTM(요구사항추적표)에서 산출물 간 추적이 가능해야 한다.
- **다이어그램**: 시스템 구조·흐름도는 Mermaid 또는 아스키 아트로 표현한다. 변환 단계에서 이미지로 렌더링될 수 있어야 한다.
- **자기완결성**: 각 산출물은 변환 시점에 외부 컨텍스트 없이 단독으로 이해 가능해야 한다. 다른 산출물 참조는 ID(`REQ-FUNC-001`)로만 한다.
- **항목 누락 금지**: reference에 정의된 필수 섹션·필수 표 컬럼은 모두 존재해야 한다. 내용이 미확정이면 `(TBD)` 마커로 명시한다.
- **간결성**: 소규모는 A4 1~3페이지, 중규모는 A4 2~10페이지 분량 목표.
- **현행성**: 기존 문서의 스타일, 용어, 표기법을 일관되게 유지한다. 한 산출물 안에서 용어가 흔들리면 변환된 산출물 품질이 떨어진다.
- **포맷·내용 분리**: 글꼴, 셀 병합, 머리말/꼬리말, 페이지 번호 같은 출력 형식은 이 단계에서 다루지 않는다. 그것은 변환 스킬의 책임이다.
