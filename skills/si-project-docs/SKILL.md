# si-project-docs — 한국 SI IT 프로젝트 산출물 작성/업데이트 스킬

## 핵심 원칙

이 스킬은 **산출물 유형별 단일 파일** 관리를 기본으로 한다.

- **기본 동작**: 기존 파일이 있으면 해당 파일을 **업데이트** (새 섹션 추가, 기존 내용 보완)
- **분리 요청 시**: 사용자가 명시적으로 도메인별 분리를 요청하면 `{문서명}_{도메인}.md` 형식으로 파일을 나눈다 (예: `API명세서_USER.md`, `API명세서_ORDER.md`)

## 씨앗 문서 활용 (프로젝트 초기)

프로젝트 최초 시작 시, `@바이브코딩용-최소요구사항정의서.md`를 씨앗(seed) 문서로 활용한다.

1. `planbot/@바이브코딩용-최소요구사항정의서.md`를 `prj/docs/`에 복사
2. 다음 항목을 우선 작성:
   - `## 0. AI 컨텍스트 스냅샷` — 전체적인 흐름 기록
   - `## 1. 프로젝트 개요` — 전체 목표 및 범위
   - `## 2.1 기술 스택 상세` — 사용 언어, 프레임워크, DB 등
3. 이 씨앗 문서를 기반으로 규모별 가이드를 참고하여 개별 산출물로 확장

## R&R 협의 가이드

문서 작성 중 불확실한 요소가 발생하면 담당자와 협의가 필요하다. 사용자에게 적절한 협의 대상을 안내한다.

| 구분              | 협의 대상 | 내용                                  |
| ----------------- | --------- | ------------------------------------- |
| **시스템 구성**   | SE 담당자 | 인프라, 서버 구조, 네트워크 등        |
| **기술 스펙**     | PL 담당자 | 라이브러리 버전, 아키텍처, 컨벤션 등  |
| **요구사항 분석** | PM 담당자 | 비즈니스 로직, 기능 범위, 우선순위 등 |

문서 내용 중 위 영역에 해당하는 불확실한 항목이 있으면, 플레이스홀더로 남기고 "이 항목은 {SE/PL/PM} 담당자와 협의가 필요합니다"라고 안내한다.

## Build 모드 연계

산출물 작성이 완료된 후 코드를 생성/수정할 때:

- 특정 코드 위치를 직접 수정하지 않는다
- **`docs/화면설계서.md` 등 문서의 수정 내용을 코드에 반영해달라고 요청**한다
- AI Agent가 관련 디렉터리의 코드를 전체적으로 파악하여 반영해야 일관성이 유지된다

### Step 3: reference 파일 읽기

대상 문서의 reference 파일을 Read 도구로 읽어서 작성 방법, 필수 섹션, 표 구조를 확인한다.
규모별 가이드에서 해당 문서의 Tier(필수/선택/생략)도 확인한다.

### Step 4: 기존 문서 확인 및 작성/업데이트

1. **기존 파일 탐색**: `docs/` 디렉터리에서 해당 문서명으로 Glob 검색
2. **파일이 있으면**: 기존 내용을 Read로 읽고, 요청된 부분만 Edit으로 업데이트
3. **파일이 없으면**: reference의 구조에 따라 새 파일을 Write로 생성

### Step 5: 버전 이력 업데이트

문서 하단의 변경 이력 테이블에 변경 내용을 추가한다.

```markdown
## 변경 이력

| 버전 | 일자       | 작성자 | 변경 내용            |
| ---- | ---------- | ------ | -------------------- |
| v1.0 | 2026-04-01 | 홍길동 | 초안 작성            |
| v1.1 | 2026-04-02 | 홍길동 | 주문 도메인 API 추가 |
```

## 도메인별 파일 분리 (선택적)

사용자가 **명시적으로** 도메인별 분리를 요청한 경우에만 적용한다.

**트리거 문구 예시:**

- "API명세서를 도메인별로 나눠줘"
- "USER랑 ORDER 설계서 분리해줘"
- "화면설계서를 기능별로 쪼개줘"

**분리 규칙:**

- 파일명: `{문서명}_{도메인}.md` (예: `API명세서_USER.md`, `화면설계서_ORDER.md`)
- 각 파일은 동일한 메타데이터 헤더를 가지되, 범위를 명시
- 목록 파일(`{문서명}_INDEX.md`)을 생성하여 분리된 파일들을 참조

**분리하지 않는 경우 (기본):**

- 하나의 파일 안에서 도메인별 섹션(`## 2.1 사용자 관리 API`, `## 2.2 주문 관리 API`)으로 구분

## 메타데이터 헤더 표준

모든 산출물은 다음 메타데이터 헤더로 시작한다:

```markdown
# {문서 제목}

| 항목       | 내용          |
| ---------- | ------------- |
| 문서명     | {문서명}      |
| 프로젝트명 | {프로젝트명}  |
| 버전       | v1.0          |
| 작성일     | {YYYY-MM-DD}  |
| 작성자     | {이름 (역할)} |
| 승인자     | {이름 (역할)} |
```

프로젝트명, 작성자 등은 기존 문서나 프로젝트 컨텍스트에서 가져온다.
정보가 없으면 `{프로젝트명}` 등 플레이스홀더로 남기고 사용자에게 알린다.

## 작성 품질 기준

- **표 중심**: 핵심 정보는 마크다운 표로 정리
- **ID 체계 유지**: 기존 문서의 ID 넘버링 규칙을 따름 (예: `REQ-FUNC-001`, `API-AUTH-001`, `SCR-DASH-001`)
- **다이어그램**: 시스템 구조, 흐름도는 아스키 아트 또는 Mermaid로 표현
- **간결성**: 소규모는 A4 1~3페이지, 중규모는 A4 2~10페이지 목표
- **현행성**: 기존 문서의 스타일, 용어, 표기법을 일관되게 유지
