# 변경관리요청서 작성 가이드

## 목적
프로젝트 수행 중 발생하는 범위·일정·비용 변경 요청을 공식 절차를 통해 승인·관리하여 무분별한 범위 확대(Scope Creep)를 방지하고 변경 이력을 추적한다.

## 작성 시점
변경 요청이 발생할 때마다 즉시 작성하여 PM의 영향도 분석 및 승인을 거친다. 변경 요청은 착수 이후 어느 단계에서든 발생할 수 있으며, 승인 없이 작업을 시작해서는 안 된다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 변경관리요청서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v0.0 ({N}건 요청 수록)} |
| 기준일 | {YYYY-MM-DD} |
| 작성자 | {작성자명 (역할)} |

## 필수 섹션 구조

변경 요청이 여러 건인 경우, 요청 건별로 아래 구조를 반복하여 하나의 문서에 수록한다. 각 요청은 `## CR-{NNN}: {제목}` 형태의 소제목으로 구분한다.

### CR-{NNN}: {변경 요청 제목}

**기본 정보 표:** 변경 요청 ID(`CR-NNN`), 요청자, 요청일, 변경 유형(범위 변경/일정 변경/비용 변경 등)을 표로 기록한다.

**변경 내용 표:** 현행 내용, 변경 요청 내용, 변경 사유를 표로 명확하게 대비하여 기술한다.

**영향도 분석 표:** 일정, 비용, 품질 각 분야에 대한 영향 내용과 영향도(높음/중간/낮음/없음)를 표로 분석한다.

**승인 결과:** 승인(✅)/반려(❌)/보류(⏸️) 결과, 승인자명, 승인일을 인라인으로 명시한다.

**조치 결과:** 개발·반영 완료 일자와 구체적인 조치 내용을 기술한다.

### 변경 요청 현황 요약
변경 유형별(기능 추가, 일정 변경, 비용 변경 등) 전체/승인/반려/보류/진행 중 건수를 집계한 표를 문서 하단에 제공한다.

### 변경 이력
버전, 변경일, 변경 내용, 작성자를 표로 관리한다.

## 작성 팁

- 영향도 분석은 변경 요청자가 아닌 PM 또는 기술 리드가 객관적으로 수행하며, 영향도가 "중간" 이상인 경우 발주처와 사전 협의를 거친다.
- 승인 결과를 기록할 때 승인자명과 승인일을 반드시 명시하여 구두 승인이 아닌 공식 기록으로 관리한다.
- 조치 결과에는 요구사항 문서(요구사항정의서 등)에 반영된 항목 ID 또는 커밋 정보를 연계하여 변경 추적성(Traceability)을 확보한다.
