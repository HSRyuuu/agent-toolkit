# 대규모 프로젝트 문서 가이드

## 적용 기준

| 구분 | 기준 |
|------|------|
| 팀 규모 | 30명 이상 (다수 팀 / PMO 조직) |
| 프로젝트 기간 | 12개월 이상 |
| 예산 | 10억 원 이상 |
| 외부 시스템 연동 | 10건 이상 |
| 적용 방법론 | 워터폴 / PMO (공식 감리 포함) |
| 승인 프로세스 | 발주처 + 감리 공식 승인 |
| 필수 산출물 | 40종 이상 |

### 문서 분류 기준

| 분류 | 설명 |
|------|------|
| 관리 문서 | 프로젝트 계획·모니터링·통제에 활용되는 문서 |
| 기술 문서 | 시스템 분석·설계·개발·테스트를 위한 기술적 산출물 |
| 운영 문서 | 시스템 운영·유지보수·교육에 활용되는 문서 |
| 계약·행정 문서 | 발주처와의 계약·협약·감리 등을 위한 공식 문서 |

## Tier 2 — 선택 문서

대규모 프로젝트에서는 아래 문서가 계약·감리·규정 요건에 따라 추가될 수 있다.

| 문서명 | 작성 조건 |
|--------|----------|
| 감리 계획서 | 행정안전부 정보화사업 또는 공공 프로젝트 |
| 보안 관리 계획서 | 개인정보 대량 처리 또는 금융·공공 서비스 |
| 교육 훈련 계획서 | 사용자 교육이 대규모로 필요한 경우 |
| 유지보수 계획서 | 운영 이관 후 장기 유지보수 계약이 있는 경우 |

## 문서 작성 사이클

```
착수/기획  →  분석  →  설계  →  구축  →  시험  →  전개  →  운영
   ↓           ↓        ↓        ↓        ↓        ↓        ↓
 사업기획서  요구사항  아키텍처  소스코드  테스트   배포계획  운영매뉴얼
 프로젝트헌장 정의서   설계서   개발가이드 결과서   릴리즈노트  장애처리
```

### 단계별 주요 흐름

```
착수/기획
  ├─ 사업 기획서 → 타당성 분석서 → SWOT → 비즈니스 모델 캔버스
  └─ 프로젝트 헌장 서명 → 공식 착수

분석
  ├─ 현행 시스템 분석서 → 요구사항 수집
  ├─ 요구사항 정의서 작성 → RTM 초안
  ├─ 보안 정책 정의서 + 시스템 구성도
  └─ [승인] 발주처 분석 완료 확인

설계
  ├─ 시스템 아키텍처 설계서 → DB 설계서 (논리→물리)
  ├─ 화면 설계서 → API 명세서 → 인터페이스 설계서
  ├─ 프로그램 사양서 + 시퀀스 다이어그램 (복잡 로직)
  ├─ 이행 계획서 (데이터 이행 해당 시)
  └─ [승인] 발주처 설계 완료 확인

구축
  ├─ 개발 환경 구축 가이드 + 코딩 표준서 배포
  ├─ 기능 단위 개발 + 단위 테스트 결과서
  └─ ERD (물리 모델) 확정

시험
  ├─ 테스트 계획서 → 테스트 케이스 작성
  ├─ 통합 테스트 수행 → 통합 테스트 결과서
  ├─ 성능 테스트 보고서 + 보안 취약점 점검 결과서
  └─ [승인] 발주처 시험 완료 확인

전개
  ├─ 배포 계획서 실행 → 릴리즈 노트 발행
  ├─ 오픈 계획서 + 비상 대책 방안 수행
  ├─ 인수인계 계획서에 따라 운영 이관
  └─ [승인] 발주처 오픈 승인

운영
  ├─ 사용자 매뉴얼 + 운영자 매뉴얼 최종본 배포
  ├─ 백업 및 복구 계획서 수립·이행
  ├─ 서비스 수준 협약서(SLA) 체결
  └─ 프로젝트 종료 보고서 작성 + 공식 완료 서명
```

## 작성 원칙

1. **전 생애주기 추적 가능성 확보** — 요구사항(REQ-) → 설계 → 개발 모듈 → 테스트 케이스(TC-)까지 RTM으로 연결하여 누락 없이 추적한다.

2. **단계별 공식 승인 의무화** — 각 단계 완료 시 발주처 서명을 받은 후 다음 단계로 진입한다. 승인 없이 다음 단계 진입 시 범위·책임 분쟁의 원인이 된다.

3. **문서 형상 관리** — 모든 문서는 Git, Confluence, SharePoint 등 형상 관리 도구로 이력 관리. 최종본은 `_FINAL` 태그로 구분. 수정 시 변경 이력 섹션에 반드시 기록.

4. **식별 코드 체계 준수** — 요구사항(REQ-FUNC-001), API(API-001), 테스트 케이스(TC-001), 결함(BUG-001), 변경 관리 요청(CR-001) 형식으로 전 문서에 일관되게 적용.

5. **문서 품질 4원칙** — 명확성(수치 명시, 모호한 표현 배제), 완전성(누락 항목 없음), 일관성(문서 간 상충 방지), 추적 가능성(RTM 연결 유지).

### 버전 관리 체계

| 버전 | 의미 |
|------|------|
| `v0.x` | 초안 / 내부 검토 중 |
| `v1.0` | 내부 승인 완료, 발주처 초도 공유 |
| `v1.x` | 소수점 수정 (내용 보완) |
| `vN.0` | 정수 변경 (대폭 변경, 공식 재승인 필요) |
