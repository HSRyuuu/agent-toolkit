# 개발환경구축가이드 작성 가이드

## 목적

프로젝트에 신규 투입되는 개발자가 로컬 개발 환경을 일관되게 구성할 수 있도록 도구 목록, 설치 절차, 코딩 표준, 브랜치 전략을 문서화한다. 팀 전체의 개발 환경 통일과 온보딩 시간 단축이 목적이다.

## 작성 시점

구축 단계 초기(개발 환경 확정 직후, 첫 번째 개발 Sprint 시작 전)에 초안을 작성하고, 도구·패키지 버전 변경 시 즉시 갱신한다.

## 필수 섹션 구조

### 1. 개발 도구 목록
프로젝트에서 사용하는 모든 개발 도구를 표로 정리한다. 도구명, 필요 버전, 용도, 설치 방법(URL 또는 명령어)을 포함한다. 버전은 최소 요구 버전 또는 권장 버전을 명시한다.

### 2. 환경 구성 절차
개발자가 순서대로 따라할 수 있는 단계별 절차를 기술한다. 각 단계는 코드 블록(bash)으로 명령어를 포함하며, 소절로 구분한다.
- **2.1 저장소 Clone**: 저장소 URL, SSH/HTTPS 방식 선택 기준
- **2.2 로컬 개발 환경 구성**: Docker Compose 또는 동등한 방법으로 전체 서비스 기동 절차, 환경 변수 파일 설정 방법
- **2.3 Frontend 개발 환경**: 의존성 설치, 개발 서버 실행, 타입 체크, 린트 명령어
- **2.4 Backend 개발 환경**: 가상 환경 생성, 의존성 설치, 개발 서버 실행, DB 마이그레이션, 테스트 실행 명령어

### 3. 의존성 관리
프로젝트에서 사용하는 주요 패키지를 레이어별로 표로 정리한다.
- **3.1 Frontend 주요 패키지**: 패키지명, 버전, 용도
- **3.2 Backend 주요 패키지**: 패키지명, 버전, 용도
- **3.3 보안 취약점 검사**: 의존성 취약점 검사 명령어와 조치 기준

### 4. 코딩 표준서
팀 내 코드 품질 통일을 위한 규칙을 정의한다.
- **4.1 Frontend 코딩 규칙**: 들여쓰기, 줄 끝, 최대 줄 길이, 파일명·함수명·상수명 규칙, import 순서
- **4.2 Backend 코딩 규칙**: 들여쓰기, 포매터, 린터, 타입 힌트 정책, 네이밍 규칙, 주석 언어
- **4.3 코드 리뷰 기준**: PR 크기 제한, 리뷰어 수, CI 통과 조건, 보호 브랜치 정책

### 5. 브랜치 전략
팀의 Git 워크플로우를 정의한다.
- **5.1 브랜치 구조**: 주요 브랜치(main, develop, feature, bugfix, hotfix 등)의 역할을 계층 구조로 표현
- **5.2 브랜치 명명 규칙**: 브랜치 유형별 명명 형식과 예시
- **5.3 PR/Merge 정책**: 브랜치 간 Merge 방식(squash/merge commit), 승인 조건

### 변경 이력
버전별 변경 내용을 추적한다.

## 작성 팁

- **환경 변수 보안**: `.env.example` 파일에 키 목록만 남기고 실제 값은 팀 내부 채널(Vault, 팀 위키 등)에서 배포한다. 문서에 실제 API 키나 시크릿을 직접 기재하지 않는다.
- **명령어 검증**: 문서에 기재된 모든 bash 명령어는 클린 환경에서 직접 실행하여 동작을 검증한 후 작성한다. 버전 불일치로 인한 오류가 흔한 온보딩 장애물이다.
- **버전 고정 vs 범위**: 패키지 버전은 `20.x LTS`처럼 범위로 표기하되, 재현 가능한 빌드가 중요한 경우 `package-lock.json` / `uv.lock` 등 lock 파일을 함께 커밋하고 그 사실을 명시한다.
