# 단위테스트결과서 작성 가이드

## 목적

개발 완료된 각 모듈(서비스, 컴포넌트, 어댑터 등)이 설계된 대로 동작하는지를 함수·클래스 단위로 검증하고, 테스트 케이스별 수행 결과와 코드 커버리지를 기록하여 구축 단계 완료의 근거로 삼는다.

## 작성 시점

구축 단계 종료 시점(단위 테스트 수행 완료 후)에 작성한다. 일반적으로 개발과 병행하여 테스트를 수행하고, 전체 결과를 취합하여 구축 단계 산출물로 제출한다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 단위테스트결과서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v1.0} |
| 테스트 기간 | {YYYY-MM-DD} ~ {YYYY-MM-DD} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {이름 (역할)} |
| 검토자 | {이름 (역할)} |

## 필수 섹션 구조

### 1. 테스트 범위
단위 테스트를 수행한 모듈 전체를 표로 나열한다. 모듈 ID(프로그램 ID), 테스트 대상 클래스·함수, 담당 개발자를 포함한다. 이 표는 단위 테스트 케이스 목록과 1:N으로 대응된다.

### 2. 테스트 케이스 및 결과
모듈별로 소절을 나누어 테스트 케이스 표를 작성한다. 각 소절의 제목은 `## 2.N {모듈명} ({모듈 ID})` 형식을 따른다. 표 컬럼은 케이스 ID, 테스트 항목, 입력값, 예상 결과, 실제 결과, 판정(Pass/Fail)을 포함한다. 정상 흐름과 예외 흐름(잘못된 입력, 권한 오류, 외부 API 오류 등) 케이스를 모두 포함한다.

### 3. 커버리지 측정
테스트 프레임워크가 출력한 커버리지 결과를 그대로 수록한다.
- **3.1 Backend 코드 커버리지**: 모듈별 구문(Stmts), 미커버(Miss), 커버리지(%) 를 코드 블록으로 표시하고, 목표 대비 달성 여부를 명시한다.
- **3.2 Frontend 코드 커버리지**: Lines, Branches, Functions, Statements 커버리지를 코드 블록으로 표시하고, 목표 대비 달성 여부를 명시한다.

### 4. 결함 목록
단위 테스트 수행 중 발견된 결함을 모두 기록한다. 결함 ID, 발견된 케이스 ID, 결함 내용, 심각도(상/중/하), 조치 여부, 조치 내용을 포함한다. 결함이 없을 경우 "발견된 결함 없음"으로 명시한다.

### 5. 검토 의견
테스트 리더 또는 QA 담당자가 종합 의견(Pass율, 커버리지 달성 여부, 미해결 결함, 후속 단계 재검증 필요 사항 등)을 자유 서술 형식으로 기재하고 서명·날짜를 남긴다.

### 변경 이력
버전별 변경 내용을 추적한다.

## 작성 팁

- **케이스 ID 체계 통일**: 케이스 ID는 `UT-{모듈약어}-{순번}` 형식으로 통일하면 결함 목록과 커버리지 리포트에서 역추적이 용이하다. 모듈약어는 테스트 범위 표의 모듈 ID와 연계한다.
- **예외 케이스 빠짐 없이**: 정상 케이스만 통과시키는 테스트는 실질적 품질 보증이 되지 않는다. 잘못된 입력값, 권한 오류, 외부 API 타임아웃·오류 등 예외 흐름을 반드시 케이스에 포함한다.
- **커버리지 캡처 방법 명시**: 커버리지 수치는 어떤 명령어로 생성한 결과인지(`pytest --cov`, `jest --coverage` 등)를 본문 또는 각주에 기재하여 재현 가능하게 한다.
