# 코딩표준서 작성 가이드

## 목적

팀 전체가 일관된 코드를 작성하여 유지보수성과 코드 리뷰 효율을 높인다. 네이밍 규칙, 포맷팅, 주석 규칙, Git 브랜치 전략, 커밋 메시지 규칙, 코드 리뷰 체크리스트를 정의함으로써 팀원 간 코드 스타일 충돌을 방지하고 온보딩 시간을 단축한다.

> **적용 기준:** 중규모 이상 프로젝트에서는 Tier 1 필수 문서로 작성하며, 소규모 프로젝트(팀원 3명 이상, 코드 리뷰 진행 시)에서는 Tier 2 선택 권장 문서로 작성한다. 소규모에서는 개발환경구축가이드의 "코딩 표준" 섹션에 통합하여 작성할 수 있다.

## 작성 시점

구축 단계 시작 시 — Sprint 1 첫날에 초안을 작성하고 팀 합의를 거쳐 확정한다. 린터·포매터 설정 파일(`.eslintrc`, `pyproject.toml` 등)과 함께 저장소에 커밋한다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 코딩표준서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v1.0} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {이름 (역할)} |
| 승인자 | {이름 (역할)} |

## 필수 섹션 구조

### 1. 네이밍 규칙

변수, 함수, 클래스, 상수, 파일명의 명명 규칙을 언어·레이어별로 정의한다. 규칙과 함께 올바른 예시(Good)와 잘못된 예시(Bad)를 병기하면 이해도가 높아진다.

### 2. 코드 포맷팅

들여쓰기, 줄 길이 제한, 중괄호 위치, 빈 줄 규칙, 줄 끝 문자를 정의한다. 자동화 도구(Prettier, Black, isort 등) 설정 파일 경로를 명시하여 수동 적용 오류를 방지한다.

### 3. 주석 규칙

함수·클래스 문서화 주석(JSDoc, docstring 등) 작성 기준과 인라인 주석 작성 지침을 정의한다. 주석 작성 언어(한국어/영어)도 명시한다.

### 4. 오류 처리 패턴

예외 처리 표준 패턴, 로그 레벨 기준(DEBUG/INFO/WARN/ERROR), 에러 응답 형식을 정의한다. 절대 해서는 안 되는 패턴(빈 catch 블록, 예외 무시 등)도 명시한다.

### 5. 보안 코딩 규칙

SQL Injection 방지(파라미터 바인딩 의무화), XSS 방지(출력 이스케이프), 입력값 검증, 시크릿 하드코딩 금지 등의 보안 필수 규칙을 기술한다.

### 6. Git 브랜치 전략

주요 브랜치 역할(main, develop, feature, bugfix, hotfix, release)과 브랜치 명명 규칙, 브랜치 간 Merge 방식(squash merge / merge commit)을 정의한다.

### 7. 커밋 메시지 규칙

커밋 메시지 형식(타입: 제목 / 본문 / 꼬리말)과 허용 타입 목록(feat, fix, docs, style, refactor, test, chore 등)을 정의한다. 예시를 2~3개 함께 제공한다.

### 8. 코드 리뷰 체크리스트

PR 제출자가 제출 전에 자가 점검해야 하는 항목과 리뷰어가 검토해야 하는 항목을 구분하여 목록으로 제공한다. CI 통과 조건, PR 크기 제한, 승인 조건도 포함한다.

### 9. 변경 이력

버전별 변경 내용을 추적한다.

## 주요 표 템플릿

### 네이밍 규칙 표

| 대상 | 규칙 | Good 예시 | Bad 예시 |
|------|------|----------|---------|
| 변수 | {camelCase \| snake_case} | {goodExample} | {BadExample} |
| 함수/메서드 | {camelCase \| snake_case} | {getUserById} | {GetUserByID} |
| 클래스 | {PascalCase} | {UserService} | {user_service} |
| 상수 | {UPPER_SNAKE_CASE} | {MAX_RETRY_COUNT} | {maxRetryCount} |
| 파일명 | {kebab-case \| snake_case} | {user-service.ts} | {UserService.ts} |
| DB 테이블/컬럼 | {snake_case} | {user_profile} | {UserProfile} |

### 코드 포맷팅 규칙 표

| 항목 | 규칙 | 비고 |
|------|------|------|
| 들여쓰기 | {공백 N칸 \| 탭} | {설정 파일 경로} |
| 최대 줄 길이 | {N자} | {설정 파일 경로} |
| 줄 끝 문자 | {LF \| CRLF} | {.gitattributes에서 강제} |
| 중괄호 위치 | {K&R \| Allman} | {예시 코드 참조} |
| 포매터 | {Prettier \| Black \| gofmt 등} | {버전 명시} |
| 린터 | {ESLint \| Flake8 \| golangci-lint 등} | {버전 명시} |

### 로그 레벨 기준 표

| 레벨 | 사용 기준 | 예시 |
|------|---------|------|
| DEBUG | 개발 시 디버깅용, 운영 환경에서는 비활성화 | {변수 값 출력, 함수 진입/이탈} |
| INFO | 정상 처리 흐름의 주요 이벤트 | {사용자 로그인 성공, 배치 작업 완료} |
| WARN | 즉각 조치는 불필요하지만 모니터링 필요 | {재시도 발생, 느린 쿼리 감지} |
| ERROR | 예외 발생 또는 처리 실패, 즉시 확인 필요 | {DB 연결 실패, API 호출 오류} |

### 브랜치 명명 규칙 표

| 유형 | 형식 | 예시 |
|------|------|------|
| 기능 개발 | `feature/{이슈ID}-{설명}` | `feature/123-user-login` |
| 버그 수정 | `bugfix/{이슈ID}-{설명}` | `bugfix/456-token-expired` |
| 긴급 수정 | `hotfix/{이슈ID}-{설명}` | `hotfix/789-null-pointer` |
| 릴리즈 | `release/v{버전}` | `release/v1.2.0` |

### 커밋 메시지 타입 표

| 타입 | 설명 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 (코드 변경 없음) |
| `style` | 포맷팅, 세미콜론 누락 등 (동작 변경 없음) |
| `refactor` | 리팩토링 (기능 변경 없음) |
| `test` | 테스트 추가 또는 수정 |
| `chore` | 빌드 설정, 패키지 관리 등 기타 변경 |

### PR/Merge 정책 표

| 항목 | 정책 |
|------|------|
| PR → develop | Squash Merge, 승인 {N}명 이상 + CI 통과 |
| develop → main | Merge Commit, 승인 {N}명 이상 + CI 통과 |
| hotfix → main | Merge Commit, 긴급 승인 {N}명 이상 |
| PR 최대 크기 | 변경 파일 {N}개 이하, 추가 라인 {N}줄 이하 |

### 코드 리뷰 체크리스트 (PR 제출자)

| 항목 | 확인 |
|------|------|
| 린터/포매터 경고 없이 통과 | ☐ |
| 단위 테스트 추가 또는 기존 테스트 통과 | ☐ |
| 시크릿·API 키 하드코딩 없음 | ☐ |
| PR 설명에 변경 내용 및 테스트 방법 기재 | ☐ |
| PR 크기가 정책 이내 (초과 시 분할) | ☐ |
| 디버그 코드(console.log, print 등) 제거 | ☐ |

### 변경 이력

| 버전 | 변경일 | 변경 내용 | 작성자 | 승인자 |
|------|--------|----------|--------|--------|
| {v1.0} | {YYYY-MM-DD} | {초안 작성} | {이름} | {이름} |

## 작성 팁

- **린터와 포매터 설정 파일을 문서보다 우선한다.** `.eslintrc`, `pyproject.toml`, `.prettierrc` 등 자동화 도구 설정 파일을 저장소에 함께 커밋하면 문서가 구식이 되어도 코드 스타일은 자동으로 강제된다. 문서에는 설정 파일의 핵심 항목만 요약하고 파일 경로를 명시한다.
- **Good/Bad 예시를 반드시 병기한다.** 규칙만 나열하면 해석이 분분해진다. 각 규칙마다 허용되는 예시와 금지되는 예시를 코드 블록으로 제시하면 모호함을 없앨 수 있다.
- **커밋 메시지 규칙은 자동 검증 도구와 연동한다.** commitlint + husky를 도입하면 규칙에 맞지 않는 커밋 메시지를 자동으로 거부할 수 있다. 문서에 도구 설치 명령어도 함께 기재한다.
- **소규모 프로젝트에서는 개발환경구축가이드에 통합한다.** 팀원이 2명 이하이거나 코드 리뷰를 진행하지 않는다면 별도 파일 대신 개발환경구축가이드의 "코딩 표준서" 섹션에 핵심 규칙만 간략히 기재한다.
