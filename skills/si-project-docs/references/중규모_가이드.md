# 중규모 프로젝트 문서 가이드

## 적용 기준

| 구분 | 소규모 | **중규모 (본 가이드)** | 대규모 |
|------|--------|----------------------|--------|
| 팀 규모 | 2~10명 | **10~30명** | 30명 이상 |
| 프로젝트 기간 | 1~3개월 | **3~12개월** | 12개월 이상 |
| 예산 | 1억 원 미만 | **1억~10억 원** | 10억 원 이상 |
| 외부 시스템 연동 | 0~2건 | **3~10건** | 10건 이상 |
| 팀 구성 | 풀스택 1~3명 | **기능별 역할 분리** | 다수 팀 / PMO |
| 적용 방법론 | 린/애자일 | **하이브리드 (Waterfall + Agile)** | 워터폴 / PMO |
| 필수 산출물 | 9종 | **15종** | 40종 이상 |

> 위 기준 중 3가지 이상 해당하면 본 가이드를 적용한다.

### 하이브리드 방법론 구조

```
[착수·분석·설계] ─── Waterfall 방식 ───────────────────────────────────
  문서 우선 작성 → 단계 게이트(Phase Gate) 승인 후 다음 단계 진입
  이해관계자 공식 승인 필수, 요구사항 변경 시 변경 관리 프로세스 적용

[구축·시험] ────── Agile Sprint 방식 ────────────────────────────────
  2주 단위 스프린트 반복, 스프린트 계획·리뷰·회고 운영
  설계 문서 기반으로 점진적 개발, 매 스프린트 기능 단위 산출물 업데이트

[전개·운영] ──── Waterfall 방식 ─────────────────────────────────────
  공식 배포 절차, 최종 승인 후 운영 이관, 운영 문서 완성
```

## Tier 2 — 권장 문서 (7종)

| 문서명 | 산출문서명 | 작성 판단 기준 | 분량 목표 |
|--------|-----------|---------------|-----------|
| WBS + 프로젝트 진행 보고서 | WBS, 프로젝트진행보고서 | 외부 발주처가 있거나 주기적 보고가 필요할 때 | A4 2~3페이지 (주간) |
| 위험·이슈 관리 대장 | 위험이슈관리대장 | 이해관계자 4명 이상이거나 리스크가 높을 때 | 표 형식 누적 관리 |
| 변경 관리 요청서 | 변경관리요청서 | 계약 기반 프로젝트이거나 범위 변경이 빈번할 때 | 건당 A4 1페이지 |
| 성능 테스트 보고서 | 성능테스트보고서 | 동시 접속자 100명 이상 또는 성능 SLA가 있을 때 | A4 3~5페이지 |
| 보안 취약점 점검 결과서 | 보안취약점점검결과서 | 개인정보 처리 또는 공공·금융 서비스일 때 | A4 3~5페이지 |
| 프로그램 사양서 | 프로그램사양서 | 복잡한 배치·알고리즘 로직이 있을 때 | 프로그램 수 × 1페이지 |
| 프로젝트 종료 보고서 | 프로젝트종료보고서 | 공식 완료 기록 및 교훈 도출이 필요할 때 | A4 5~8페이지 |

### Tier 2 판단 흐름

```
외부 발주처 존재?           → WBS + 프로젝트 진행 보고서 필수 작성
이해관계자 4명 이상?        → 위험·이슈 관리 대장 작성
계약 기반 프로젝트?         → 변경 관리 요청서 작성
동시 접속 100명 이상 예상?  → 성능 테스트 보고서 작성
개인정보 처리 / 공공 서비스? → 보안 취약점 점검 결과서 작성
복잡 배치 로직 존재?        → 프로그램 사양서 작성
공식 완료 기록 필요?        → 프로젝트 종료 보고서 작성
```

## 문서 작성 사이클

```
Phase 0  착수 (2~4주)
  ├─ 킥오프 회의
  ├─ 사업 기획서 확정
  ├─ 프로젝트 헌장 작성·서명
  └─ [Phase Gate 0] 착수 승인

Phase 1  분석 (4~6주)
  ├─ 현행 시스템 분석
  ├─ 요구사항 수집·요구사항 정의서 작성
  ├─ 요구사항 추적표(RTM) 초안 작성
  ├─ 시스템 구성도 확정
  └─ [Phase Gate 1] 분석 완료 승인 (발주처 서명)

Phase 2  설계 (4~6주)
  ├─ 시스템 아키텍처 설계서 작성
  ├─ 데이터베이스 설계서 작성 (논리·물리)
  ├─ 화면 설계서 작성
  ├─ API 명세서 초안
  ├─ 인터페이스 설계서 작성
  └─ [Phase Gate 2] 설계 완료 승인 (발주처 서명)

Phase 3  구축 (스프린트 반복, 6~16주)
  ├─ 개발 환경 구축 가이드 완성
  ├─ 코딩 표준서 배포
  ├─ [Sprint 1~N] 기능 단위 개발·단위 테스트
  │     - 스프린트 계획 → 개발 → 데모 → 회고
  │     - 스프린트당 화면 설계서·API 명세서 업데이트
  └─ [Phase Gate 3] 개발 완료 확인

Phase 4  시험 (4~6주)
  ├─ 테스트 계획서 확정
  ├─ 통합 테스트 수행·통합 테스트 결과서 작성
  ├─ 성능 테스트 보고서·보안 취약점 점검 결과서 (Tier 2, 해당 시)
  └─ [Phase Gate 4] 테스트 완료 승인 (발주처 서명)

Phase 5  전개·운영 이관 (2~4주)
  ├─ 배포·오픈·인수인계 통합 계획서 실행
  ├─ 사용자 매뉴얼·운영자 매뉴얼 최종 완성
  ├─ 안정화 운영 (2~4주)
  └─ [Phase Gate 5] 최종 인수 서명·프로젝트 종료 보고서
```

### Phase Gate 승인 기준

| Gate | 단계 | 승인 조건 | 승인자 |
|------|------|-----------|--------|
| Gate 0 | 착수 완료 | 프로젝트 헌장 서명, 팀 구성 완료 | PM·발주처 |
| Gate 1 | 분석 완료 | 요구사항 정의서 승인, RTM 초안 완성 | PM·발주처 |
| Gate 2 | 설계 완료 | 아키텍처·DB 설계서·화면 설계서·API 명세서 승인 | PM·발주처 |
| Gate 3 | 개발 완료 | 단위 테스트 Pass율 90% 이상, 코드 리뷰 완료 | PM |
| Gate 4 | 시험 완료 | Critical/High 결함 0건, 통합 테스트 결과서 승인 | PM·발주처 |
| Gate 5 | 최종 인수 | 인수인계 완료, 안정화 기간 종료 | PM·발주처 |

### 스프린트 운영 기준 (구축·시험 단계)

| 항목 | 기준 |
|------|------|
| 스프린트 주기 | 2주 (10 영업일) |
| 스프린트 계획 | 스프린트 첫째 날, 2시간 이내 |
| 데일리 스탠드업 | 매일 15분, 완료·예정·블로커 3가지 보고 |
| 스프린트 데모 | 스프린트 마지막 날, 기능 시연 (발주처 참여 권장) |
| 스프린트 회고 | 데모 후, 30~60분 |
| 문서 업데이트 | 스프린트 종료 시 화면 설계서·API 명세서 최신화 |

## 작성 원칙

1. **하이브리드 방법론 준수** — 착수·분석·설계는 Waterfall(문서 선행 + 단계 게이트 승인), 구축·시험은 Agile Sprint로 운영한다.

2. **Phase Gate 전 산출물 완성도 확인** — 핵심 섹션 누락 또는 미승인 시 Gate 통과 불가. 일부 미완성 시 `[TBD]` 표시 후 조건부 통과.

3. **변경 관리 원칙** — Gate 1(분석 완료) 이후 범위 변경은 반드시 변경 관리 요청서로 접수하고 영향도 분석(일정·비용·품질) 후 PM 승인. 승인 후 요구사항 정의서·RTM·설계서에 즉시 반영.

4. **문서 표준화** — 모든 문서에 표지(문서명, 버전, 작성일, 작성자, 검토자, 승인자) 통일. 식별 코드 체계: 요구사항(REQ-), API(API-), 테스트 케이스(TC-), 결함(BUG-), 변경 관리(CR-).

5. **PMO-Lite 운영** — 전담 PMO 없이 PM이 계획 관리·품질 관리·리스크 관리·변경 관리·의사소통 관리를 겸임. 주간 WBS 업데이트 및 격주 프로젝트 진행 보고서 발송이 핵심.

### PMO-Lite 운영 방안

전담 PMO 없이 PM이 겸임하되, 다음 기능 영역을 분담한다.

| 기능 영역 | 담당 | 주요 활동 |
|-----------|------|----------|
| 일정 관리 | PM | WBS 갱신, 마일스톤 점검, 주간 보고 |
| 품질 관리 | QA/개발 리드 | 코드 리뷰, 테스트 수행, 결함 추적 |
| 변경 관리 | PM | CR 접수·영향 분석·승인, 변경 이력 관리 |
| 위험 관리 | PM + 아키텍트 | 위험 식별·평가, 대응 계획, 주간 리뷰 |
| 형상 관리 | 개발 리드 | Git 브랜치 전략, 문서 버전 관리, 릴리즈 태깅 |

## 소규모·중규모·대규모 산출물 3단계 비교표

| 단계 | 소규모 (9종) | 중규모 (15종) | 대규모 (40종+) |
|------|-------------|--------------|----------------|
| **착수/기획** | 프로젝트 헌장 | 사업 기획서 + 프로젝트 헌장 | 사업 기획서 + 프로젝트 헌장 + 타당성 분석서 + SWOT 분석서 + 비즈니스 모델 캔버스 |
| **분석** | 요구사항 정의서 | 요구사항 정의서 + 요구사항 추적표(RTM) + 시스템 구성도 | + 현행 시스템 분석서 + 보안 정책 정의서 |
| **설계** | 화면 설계서 + 데이터베이스 설계서 | + 시스템 아키텍처 설계서 + API 명세서 + 인터페이스 설계서 | + 프로그램 사양서 + 시퀀스 다이어그램 + 이행 계획서 |
