# 소규모 프로젝트 문서 가이드

## 적용 기준

| 구분 | 기준 |
|------|------|
| 팀 규모 | 2 ~ 10명 |
| 프로젝트 기간 | 1 ~ 3개월 (스프린트 1 ~ 6회) |
| 예산 | 1억 원 미만 |
| 외부 시스템 연동 | 0 ~ 2건 |
| 발주 형태 | 사내 자체 개발 또는 소규모 외주 |
| 적용 방법론 | 린 / 애자일 |

> 위 기준 중 2가지 이상 해당하면 본 가이드를 적용한다.

## Tier 1 — 필수 문서 (9종)

| # | 문서명 | 산출문서명 | 작성 시점 | 분량 목표 | 소유자 |
|---|--------|-----------|-----------|-----------|--------|
| 1 | 프로젝트 헌장 | 프로젝트헌장 | 착수 첫날 (Sprint 0) | A4 1페이지 | PM / 기획자 |
| 2 | 요구사항 정의서 | 요구사항정의서 | Sprint 0 (착수 준비) | A4 2~3페이지 | PM / 기획자 |
| 3 | 화면 설계서 | 화면설계서 | 설계 단계 (Sprint 1 시작 전) | 화면 수 × 0.5~1페이지 | PM / 기획자 |
| 4 | 데이터베이스 설계서 | 데이터베이스설계서 | 설계 단계 (화면 설계서 완료 후) | ERD 1장 + A4 1~2페이지 | 개발 리드 |
| 5 | 개발 환경 구축 가이드 | 개발환경구축가이드 | 구축 단계 시작 시 (Sprint 1 첫날) | A4 1~2페이지 | 개발 리드 |

| 9 | 회의록 | 회의록 | 매 스프린트 계획·리뷰 후 즉시 | 스프린트당 A4 0.5~1페이지 | 스프린트 서기 (돌아가며) |

## Tier 2 — 선택 권장 문서 (8종)

| 문서명 | 작성 기준 / 조건 |
|--------|-----------------|
| API 명세서 | 외부 시스템 연동 또는 프론트-백 분리 개발 시 |
| 코딩 표준서 | 팀원 3명 이상이고 코드 리뷰를 진행할 때 |
| 위험·이슈 관리 대장 | 이해관계자가 많거나 외주 계약이 있을 때 |
| 변경 관리 요청서 | 범위 변경이 빈번하거나 계약 기반 프로젝트일 때 |
| 릴리즈 노트 | 외부 사용자 대상 공개 서비스이거나 버전 관리가 필요한 경우 |
| 운영자 매뉴얼 | 운영 담당자가 개발팀이 아닌 별도 인원일 때 |
| 프로젝트 진행 보고서 | 외부 발주처에 주기적 보고가 필요할 때 |
| 프로젝트 종료 보고서 | 공식 프로젝트 완료 기록이 필요한 경우 |

### Tier 2 판단 흐름

```
외부 시스템 연동 있음?     → API 명세서 작성
팀원 3명 이상?             → 코딩 표준서 작성
외부 발주처 있음?          → 프로젝트 진행 보고서 + 변경 관리 요청서 작성
공개 서비스 출시?          → 릴리즈 노트 작성
운영 담당 분리?            → 운영자 매뉴얼 작성
프로젝트 공식 완료 필요?   → 프로젝트 종료 보고서 작성
```

## 문서 작성 사이클

```
Sprint 0 (착수 준비, 1~3일)
  └─ 프로젝트 헌장 작성·승인
  └─ 요구사항 정의서 초안 (v1.0)
  └─ 개발 환경 구축 가이드 초안
  └─ 화면 설계서 초안 (흐름 스케치)
  └─ ERD 초안

Sprint 1 ~ N (개발 반복)
  └─ [Sprint 시작] 스프린트 계획 → 회의록 작성
  └─ [Sprint 중]   화면 설계서 / 데이터베이스 설계서 점진적 업데이트
  └─ [Sprint 중]   요구사항 정의서 변경 발생 시 변경 이력 기록
  └─ [Sprint 종료] 완료 항목·이슈·결정사항 회의록 기록

Sprint 마지막 (배포 전)
  └─ QA 체크리스트 작성 및 수행
  └─ 배포 계획서 작성 및 수행

배포 후 (운영)
  └─ 사용자 매뉴얼 최종 완성
  └─ Tier 2 해당 문서 작성 (릴리즈 노트, 프로젝트 종료 보고서 등)
```

### 스프린트별 문서 작성 책임표

| 역할 | 작성 문서 |
|------|-----------|
| PM / 기획자 | 프로젝트 헌장, 요구사항 정의서, 화면 설계서, 회의록, 사용자 매뉴얼 |
| 개발 리드 | 데이터베이스 설계서, 개발 환경 구축 가이드, 배포 계획서, (API 명세서) |
| 개발자 전원 | 개발 환경 구축 가이드 보완, 코딩 표준서 준수 |
| QA / 팀원 | QA 체크리스트 수행 및 결함 기록 |

### Sprint 0 핵심 활동

| 활동 | 산출물 | 소요 시간 |
|------|--------|-----------|
| 킥오프 회의 | 프로젝트 헌장 초안 | 2시간 |
| 요구사항 워크숍 | 요구사항 정의서 v1.0 | 4~8시간 |
| 기술 스택 결정 | 개발 환경 구축 가이드 초안 | 2시간 |
| 개발 환경 세팅 | 팀 전원 로컬 환경 구동 확인 | 2~4시간 |
| 화면 흐름 스케치 | 화면 설계서 초안 | 2~4시간 |
| DB 초안 설계 | ERD 초안 | 2시간 |

## 작성 원칙

1. **MVD(Minimum Viable Documentation)** — 실제 업무에 쓰이지 않는 문서는 작성하지 않는다. 문서 한 건은 A4 기준 1~3페이지를 목표로 한다.

2. **3회 반복의 법칙** — 같은 내용을 세 번 이상 구두로 설명하게 되면 문서로 작성하고 링크를 공유한다.

3. **즉시 문서화** — 회의 직후 10분 이내 결정 사항 기록, 기능 완료 직후 환경 가이드 업데이트, 버그 수정 직후 QA 체크리스트 항목 추가.

4. **불완전해도 공유** — `[DRAFT]` 태그를 붙여서라도 빨리 공유한다. 완성도 70%의 문서가 제때 공유되는 것이 100% 문서가 늦게 공유되는 것보다 낫다.

5. **문서 소유자 지정** — 각 문서는 반드시 1명의 소유자(Owner)를 지정한다. 소유자가 없는 문서는 빠르게 낡아 신뢰를 잃는다.

### 버전 관리 규칙

| 버전 | 의미 |
|------|------|
| `v0.1` | 초안 (내부 검토 중) |
| `v1.0` | 팀 내 승인 완료 |
| `v1.x` | 소규모 수정 (내용 보완) |
| `v2.0` | 대폭 변경 (범위·방향 변경) |

## 대규모 vs 소규모 산출물 비교표

| 단계 | 대규모 산출물 | 소규모 대응 문서 | 간소화 방식 |
|------|--------------|-----------------|-------------|
| **착수/기획** | 사업 기획서, 프로젝트 헌장, 타당성 분석서, SWOT 분석서, 비즈니스 모델 캔버스 | 프로젝트 헌장 | 5종 → 1종 통합 |
| **분석** | 요구사항 정의서, RTM, 현행 시스템 분석서, 보안 정책 정의서, 시스템 구성도 | 요구사항 정의서 | 5종 → 1종 통합 |
| **설계** | 시스템 아키텍처 설계서, 데이터베이스 설계서, 화면 설계서, API 명세서, 인터페이스 설계서, 프로그램 사양서, 시퀀스 다이어그램 | 화면 설계서 + 데이터베이스 설계서 + (API 명세서 선택) | 7종 → 2~3종 |
| **구축** | 개발 환경 구축 가이드, 코딩 표준서, ERD(물리), 단위 테스트 결과서 | 개발 환경 구축 가이드 | 4종 → 1종 |
| **시험** | 테스트 계획서, 테스트 케이스, 통합 테스트 결과서, 성능 테스트 보고서, 보안 취약점 점검 결과서, QA 체크리스트 | QA 체크리스트 | 6종 → 1종 |
| **전개** | 배포 계획서, 릴리즈 노트, 오픈 계획서, 비상 대책 방안, 인수인계 계획서 | 배포 계획서 + (릴리즈 노트 선택) | 5종 → 1~2종 |
| **운영** | 사용자 매뉴얼, 운영자 매뉴얼, 백업 및 복구 계획서, 서비스 수준 협약서(SLA) | 사용자 매뉴얼 | 4종 → 1종 |
| **관리** | WBS, 위험·이슈 관리 대장, 변경 관리 요청서, 프로젝트 진행 보고서, 회의록, 프로젝트 종료 보고서 | 회의록 + (필요 시 선택) | 6종 → 1~3종 |

> **요약:** 대규모 기준 40종 이상 → 소규모 **Tier 1 필수 9종 + Tier 2 선택 최대 8종**
