# 요구사항정의서 작성 가이드

## 목적

프로젝트에서 발주처·이해관계자가 요구하는 기능적·비기능적 요구사항을 체계적으로 정의하고, 개발 범위와 기준을 확정한다. 요구사항 변경 이력을 관리하여 분쟁 발생 시 근거 문서로 활용한다.

## 작성 시점

분석 단계 초기에 이해관계자 인터뷰·설문을 완료한 후 작성하며, 발주처 검토·승인을 거쳐 확정한다. 요구사항 변경이 발생할 때마다 버전을 갱신한다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 요구사항정의서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v1.0} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {작성자명} ({역할}) |
| 승인자 | {PM명} (PM), {발주처 담당자명} (발주처) |

## 필수 섹션 구조

### 1. 프로젝트 개요
프로젝트명, 목적, 배경, 전체 범위를 표로 정리한다.

#### 1.1 메뉴 구성 (Depth)
시스템의 전체 메뉴 트리를 코드 블록(들여쓰기 ASCII 트리)으로 표현한다. 1Depth~3Depth까지 계층적으로 나열하여 개발 범위를 가시화한다.

### 2. 이해관계자 식별
요구사항 출처가 되는 이해관계자(발주처, 현업, 운영팀, 개발사 등)를 나열하고, 각 이해관계자별 수용 여부와 책임자를 명시한다.

### 3. 기능적 요구사항
업무 도메인별 소절(3.1, 3.2, …)로 분리하여 요구사항을 나열한다. 각 소절은 독립된 표로 구성하며, 요구사항 ID(REQ-FUNC-XXX) 체계를 일관되게 사용한다.

### 4. 비기능적 요구사항
성능·가용성·보안·사용성·유지보수성 등 비기능 항목을 표 하나로 통합하여 정리한다. 요구사항 ID(REQ-NFR-XXX)와 측정 가능한 기준값을 반드시 포함한다.

### 5. 제약 사항
예산, 일정, 기술 스택, 법규, 외부 연동, 보안 등 개발에 영향을 주는 제약 조건을 표로 정리한다.

### 6. 요구사항 변경 이력
버전별 변경 내용, 변경자, 승인자를 기록한다.

## 주요 표 템플릿

### 이해관계자 표

| 이해관계자 | 요구 출처 | 수용 여부 | 책임자 |
|-----------|---------|---------|--------|
| {조직명 + 부서명} | {사업 기획서 / 인터뷰 / 설문 등} | {전체 수용 / 일부 수용 / 미수용} | {이름 + 직책} |

### 기능 요구사항 표

| 요구 ID | 기능명 | 상세 설명 | 우선순위 | 담당자 | 상태 |
|---------|--------|---------|---------|--------|------|
| REQ-FUNC-{NNN} | {기능명} | {기능 상세 설명} | {상/중/하} | {담당자명} | {확정/검토중/보류} |

### 비기능 요구사항 표

| 요구 ID | 분류 | 요구사항 내용 | 측정 기준 |
|---------|------|------------|---------|
| REQ-NFR-{NNN} | {성능/가용성/보안/사용성/유지보수성} | {요구사항 설명} | {구체적 수치 또는 기준} |

### 제약 사항 표

| 구분 | 제약 내용 |
|------|---------|
| {예산/일정/기술/법규/외부연동/보안} | {제약 내용 구체적 기술} |

### 변경 이력 표

| 버전 | 변경일 | 변경 내용 | 변경자 | 승인자 |
|------|--------|---------|--------|--------|
| {v1.0} | {YYYY-MM-DD} | {변경 내용 요약} | {작성자명} | {승인자명 또는 -} |

## 작성 팁

- 요구사항 ID(REQ-FUNC-XXX, REQ-NFR-XXX)는 프로젝트 전체에서 고유하게 유지한다. RTM과 연계되므로 한 번 부여한 ID는 변경하지 않는다.
- 우선순위(상/중/하)는 발주처와 합의하여 결정하고, 우선순위 근거를 별도 메모 또는 회의록에 남겨 이후 이견 발생을 예방한다.
- 비기능 요구사항은 "빠르게", "안전하게"처럼 모호한 표현 대신 "2초 이내", "99.5% 이상"처럼 측정 가능한 수치로 기술한다.
