# 시스템구성도 작성 가이드

## 목적

신규 시스템의 전체 인프라 구성(서버, 네트워크, 클라우드 서비스, 보안 계층)을 시각적으로 정의하여, 아키텍트·개발자·발주처가 시스템 전체 구조를 동일하게 이해하도록 한다.

## 작성 시점

분석 단계 후반부에 요구사항이 확정된 이후 아키텍트와 인프라 담당자가 공동 작성한다. 설계 단계 진입 전 발주처 승인을 완료한다.

## 필수 섹션 구조

### 1. 전체 시스템 구성

#### 1.1 시스템 구성 다이어그램
인터넷(사용자)에서 시작하여 CDN/WAF → 로드밸런서 → 애플리케이션 서버 → 데이터 계층(DB, 캐시, 파일 저장) → 외부 연동(API 게이트웨이 등) 순으로 레이어를 내려가는 ASCII 박스 다이어그램으로 표현한다. 각 블록에 서브넷 유형(Public/Private/Isolated)을 명시한다.

### 2. 서버 구성

#### 2.1 서버 목록 및 역할
시스템을 구성하는 모든 서버(컨테이너 포함)를 표로 나열한다. 서버 구분, 서비스명, 인스턴스 유형·사양, 수량, 역할을 포함한다.

### 3. 네트워크 구성

#### 3.1 VPC 구성
VPC 전체 CIDR과 각 서브넷(퍼블릭/프라이빗/격리 등)의 CIDR, 배치 목적을 표로 정리한다. Multi-AZ 구성 여부를 명시한다.

#### 3.2 망 분리 정책
계층별(CDN·ALB·애플리케이션·DB·외부연동) 외부 접근 허용 여부와 내부 통신 규칙, 허용 포트를 표로 정의한다.

### 4. 인프라 사양

클라우드 플랫폼, 컨테이너 오케스트레이션, OS, DB 버전, 모니터링·로그·알림·도메인·SSL 등 인프라 전반의 기술 스택을 항목-사양 형식의 표로 정리한다.

### 5. 클라우드 구성 상세

#### 5.1 클라우드 서비스 목록
사용하는 모든 클라우드 서비스(관리형 포함)를 나열하고, 각 서비스의 용도와 주요 적용 설정(Auto Scaling 기준, 백업 주기, 암호화 여부 등)을 표로 기술한다.

### 변경 이력

버전별 변경 내용, 작성자, 승인자를 기록한다.

## 작성 팁

- 시스템 구성 다이어그램은 실제 네트워크 계층(Public/Private/Isolated Subnet)을 박스로 구분하여 보안 경계를 시각적으로 명확히 표현한다. 텍스트 기반 ASCII 다이어그램으로도 충분하며, 별도 다이어그램 도구(draw.io 등)를 사용하는 경우 이미지 파일 경로를 함께 기재한다.
- 서버 목록에는 HA(고가용성) 구성 여부와 Auto Scaling 정책을 명시하여 운영 안정성 검토 근거로 활용한다.
- 클라우드 서비스 목록은 나중에 인프라 비용 산정 및 운영 매뉴얼 작성 시 기초 자료가 되므로, 관리형 서비스(CDN, S3 등)도 빠짐없이 포함한다.
