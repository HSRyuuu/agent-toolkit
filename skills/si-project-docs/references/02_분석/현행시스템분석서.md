# 현행시스템분석서 작성 가이드

## 목적

신규 시스템 구축 전, 현재 운영 중인 시스템과 업무 프로세스를 파악하여 문제점과 개선 방향을 도출한다. AS-IS/TO-BE 비교를 통해 프로젝트의 도입 필요성과 방향성을 공식 문서로 확보한다.

## 작성 시점

분석 단계 초기에 이해관계자 인터뷰와 현행 시스템 조사를 완료한 직후 작성한다. 요구사항정의서 작성 전 선행 완료하여 개선 방향이 요구사항에 반영될 수 있도록 한다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 현행시스템분석서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v1.0} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {작성자명} ({역할}) |
| 승인자 | {PM명} (PM), {발주처 담당자명} (발주처) |

## 필수 섹션 구조

### 1. 현행 시스템 구성

#### 1.1 현재 운영 중인 시스템 목록
현재 업무에 사용 중인 시스템들을 표로 나열한다. 시스템명, 용도, 플랫폼 유형(SaaS/온프레미스/로컬 등), 운영 기간, 담당 부서를 포함한다.

#### 1.2 현행 시스템 구성도
ASCII 다이어그램(코드 블록)으로 시스템 간 연결 관계와 데이터 흐름을 시각화한다. 사용자 → 각 시스템 간의 관계를 트리 또는 흐름도 형태로 표현한다.

### 2. 업무 프로세스 현황 (AS-IS)

주요 업무 흐름별 소절(2.1, 2.2, …)로 분리하여 현행 수행 절차를 번호 목록 + ASCII 흐름 형식으로 기술한다. 각 단계에서 발생하는 비효율·문제점을 주석으로 병기하면 효과적이다.

### 3. 문제점 및 개선 사항

현행 업무의 문제 영역(문서 작성 효율, 버전 관리, 품질 일관성, 요구사항 추적 등)별로 현행 문제점과 개선 방향을 표로 정리한다. 번호를 부여하여 이후 TO-BE 목표와 연결한다.

### 4. TO-BE 목표

AS-IS와 TO-BE를 항목별로 대비하는 2열 표로 작성한다. 신규 시스템이 해결하는 항목(문서 작성 방식, 버전 관리, 승인 프로세스, 협업 환경 등)을 빠짐없이 포함한다.

### 변경 이력

버전별 변경 내용, 작성자, 승인자를 기록한다.

## 주요 표 템플릿

### 현행 시스템 목록 표

| 시스템명 | 용도 | 플랫폼 | 운영 기간 | 담당 부서 |
|---------|------|--------|---------|---------|
| {시스템명} | {용도 설명} | {SaaS / 온프레미스 / 로컬 파일 등} | {YYYY년~} | {부서명} |

### 문제점 및 개선 사항 표

| 번호 | 문제 영역 | 현행 문제점 | 개선 방향 |
|------|---------|----------|---------|
| {1} | {문제 영역} | {구체적 문제 설명} | {개선 방향 설명} |

### AS-IS / TO-BE 비교 표

| 항목 | AS-IS (현행) | TO-BE (목표) |
|------|------------|------------|
| {비교 항목} | {현행 방식 설명} | {목표 방식 설명} |

### 변경 이력 표

| 버전 | 변경일 | 변경 내용 | 작성자 | 승인자 |
|------|--------|----------|--------|--------|
| {v1.0} | {YYYY-MM-DD} | {변경 내용 요약} | {작성자명} | {승인자명 또는 -} |

## 작성 팁

- 문제점 번호와 TO-BE 항목을 대응시켜 "문제 N → 개선 방향"의 인과관계가 문서 내에서 일관되게 흐르도록 작성한다.
- 현행 시스템 구성도는 실제 데이터 흐름 방향(화살표)을 ASCII로 표현하면 발주처와의 현황 공유가 원활해진다.
- 인터뷰·설문 결과를 근거로 문제점을 기술하고, 가능하다면 공수 비율·오류 발생 빈도 등 수치를 포함하여 개선 필요성을 객관화한다.
