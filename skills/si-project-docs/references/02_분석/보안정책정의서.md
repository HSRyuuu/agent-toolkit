# 보안정책정의서 작성 가이드

## 목적

시스템 전반에 적용할 보안 원칙, 인증·권한 정책, 데이터 보호 기준, 네트워크 보안 규칙, 법규 준수 사항을 공식화한다. 개발 초기에 확정하여 설계·개발 전 단계에 보안 요구사항이 일관되게 반영되도록 한다.

## 작성 시점

분석 단계에서 시스템구성도와 요구사항정의서(보안 관련 비기능 요구사항)가 확정된 이후 아키텍트와 BA가 공동 작성한다. 설계 단계 진입 전 발주처 승인을 완료한다.

## 필수 섹션 구조

### 1. 보안 정책 방향

#### 1.1 기본 방침
시스템에 적용할 핵심 보안 원칙(최소 권한, 개인정보 최소화, 암호화 우선, 접근 로그 관리, 보안 by Design 등)을 원칙명-내용 형식의 표로 정의한다. 굵은 글씨(**원칙명**)로 강조한다.

#### 1.2 준수 법령 및 기준
적용 대상 법령(개인정보보호법 등)과 보안 프레임워크(OWASP Top 10, NIST CSF 등)를 글머리 목록으로 나열한다. 법령 개정 연도를 괄호로 명시한다.

### 2. 인증·권한 정책

#### 2.1 사용자 인증 방식
인증 방식, 토큰 유효시간, 비밀번호 복잡도·변경 주기, 로그인 실패 잠금 기준, MFA, SSO 여부 등 인증 관련 전체 정책을 항목-정책 형식의 표로 정의한다.

#### 2.2 역할 기반 접근 제어 (RBAC)
시스템의 역할 목록(관리자, PM, 작성자, 개발자, 뷰어, 발주처 등)과 각 역할의 권한 범위를 표로 정의한다. 역할명은 **굵은 글씨**로 강조한다.

### 3. 데이터 보안

#### 3.1 중요 데이터 분류
데이터를 등급(1등급 기밀~4등급 공개 등)으로 분류하고, 해당 데이터 예시와 보호 수준(암호화 필수, 접근 로그 필수, 일반 보호 등)을 표로 정의한다.

#### 3.2 암호화 기준
저장 데이터, 전송 데이터, 비밀번호, API 키, 백업 데이터 등 암호화 대상별로 암호화 방식과 적용 범위를 표로 정의한다.

#### 3.3 마스킹 대상
외부 API 전송 시 마스킹 처리해야 하는 개인정보 항목(주민등록번호, 계좌번호, 전화번호, 이메일 등)과 마스킹 방식을 표로 정의한다. 어떤 외부 API에 적용되는지 도입부에 명시한다.

### 4. 네트워크 보안

#### 4.1 네트워크 구성 원칙
망 분리 구성(서브넷 계층), 방화벽 정책, HTTPS 강제, HSTS, VPN 접근 요건, DDoS 방어 수단을 항목-정책 형식의 표로 정의한다.

#### 4.2 허용 포트 정책
구간별(인터넷→ALB, ALB→애플리케이션, 애플리케이션→DB, 애플리케이션→외부 API) 허용 포트와 접근 제한 조건을 표로 정의한다.

### 5. 법규 준수

적용 법령·기준별로 구체적인 준수 내용과 담당자를 표로 정의한다.

### 변경 이력

버전별 변경 내용, 작성자, 승인자를 기록한다.

## 작성 팁

- 데이터 분류 등급과 암호화 기준 표를 연결하여 "N등급 데이터는 암호화 방식 X를 적용한다"는 대응 관계가 문서 내에서 일관되게 유지되도록 작성한다.
- LLM·생성 AI 등 외부 API를 사용하는 경우, 3.3 마스킹 대상 섹션에서 어떤 API에 어떤 개인정보를 전송하지 않는지를 구체적으로 명시하여 개인정보보호법 준수 근거로 활용한다.
- 보안 정책은 개발 완료 후가 아닌 설계 단계 진입 전에 확정해야 한다. 정책 미확정 시 인증 방식, 토큰 구조, DB 암호화 설계가 뒤틀릴 수 있으므로 반드시 선행 승인한다.
