# 요구사항추적표(RTM) 작성 가이드

## 목적

요구사항정의서에 등록된 모든 요구사항이 설계·개발·테스트 단계에 빠짐없이 반영되었는지 추적한다. 요구사항 누락 방지, 변경 영향 분석, 인수 테스트 기준 수립에 활용된다.

## 작성 시점

초안은 설계 단계 완료 후(설계·개발 연계 시점) 작성하고, 테스트 완료 후 충족 여부를 확정하여 최종본으로 갱신한다. 요구사항이 추가·변경될 때마다 버전을 갱신한다.

## 필수 섹션 구조

### 1. 개요
본 문서의 목적(요구사항의 설계·개발·테스트 반영 여부 추적)을 1~2문장으로 기술한다. 최종 충족 여부 확정 시점(인수 테스트 완료 후)을 명시한다.

### 2. 추적 현황

#### 2.1 기능 요구사항 추적
요구사항정의서의 모든 기능 요구사항(REQ-FUNC-XXX)에 대해 설계 문서 참조, 개발 모듈 참조, 테스트 케이스 참조, 충족 여부를 표로 매핑한다. 한 행에 하나의 요구사항을 기재한다.

#### 2.2 비기능 요구사항 추적
모든 비기능 요구사항(REQ-NFR-XXX)에 대해 설계 문서 참조, 테스트 케이스 참조, 충족 여부를 표로 매핑한다. 기능 요구사항 추적 표에서 개발 모듈 참조 열을 제외한 형태로 구성한다.

### 3. 미충족 요구사항 현황
충족되지 않은 요구사항을 별도 표로 나열하고 미충족 사유, 보완 계획, 담당자를 기재한다. 모두 충족된 경우 표를 유지하되 인용 블록(`>`)으로 "미충족 요구사항 없음" 메모를 남긴다.

### 4. 추적 요약
기능 요구사항과 비기능 요구사항별로 전체 건수, 충족, 미충족, 부분충족, 충족률을 집계 표로 정리한다. 합계 행을 **굵은 글씨**로 강조한다.

### 변경 이력
버전별 변경 내용, 작성자, 승인자를 기록한다.

## 작성 팁

- 설계 문서 참조·개발 모듈 참조·테스트 케이스 참조에는 해당 문서 내 정확한 ID를 기재한다. "해당 없음"으로 비워두면 추적이 단절되므로, 참조 ID 체계를 프로젝트 초기에 팀 전체가 합의하여 통일한다.
- 미충족 요구사항이 0건이더라도 섹션 3을 삭제하지 말고 "미충족 요구사항 없음" 메모를 남긴다. 감리·검수 시 해당 섹션의 존재 자체가 점검 대상이 된다.
- 충족률은 인수 테스트 완료 후에만 100%로 기재한다. 테스트 진행 중인 경우 "진행 중" 또는 현재 충족된 비율을 기재하여 실제 상태를 정확히 반영한다.
