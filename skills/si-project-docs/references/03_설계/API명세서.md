# API 명세서 작성 가이드

## 목적
시스템이 외부(Frontend, 타 시스템)에 제공하는 모든 REST API의 엔드포인트, 요청/응답 구조, 오류 코드를 정의한다.
개발자 간 협업 기준을 통일하고, 프론트엔드·백엔드 병행 개발의 계약 문서 역할을 한다.

## 작성 시점
설계 단계 중반 — 시스템 아키텍처 설계서 및 데이터베이스 설계서 초안 완료 후 작성하며, 화면 설계서와 병행하여 작성한다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | API명세서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v0.1} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {이름 (역할)} |
| 승인자 | {이름 (역할)} |

---

## 필수 섹션 구조

### 1. API 목록
전체 API를 한눈에 파악할 수 있는 요약 목록. API ID 체계, HTTP 메서드, URL, 설명, 인증 요부를 기재한다.

### 2. API 상세 명세
각 API별 하위 절(###)로 구성하며, 다음 항목을 순서대로 기술한다.
- 요청 예시 (메서드 + URL + 주요 헤더)
- Path / Query Parameters 표
- Request Body 예시 JSON + 필드 정의 표
- Response 예시 JSON (성공 케이스, 상태 코드 명시)
- 비고 (비동기 처리, 폴링/WebSocket 안내 등)

### 3. 공통 응답 구조
성공 응답(`data`, `meta` 구조)과 오류 응답(`error.code`, `error.message`, `error.detail` 구조)의 공통 형식을 정의한다.

### 4. 오류 코드 정의
시스템 전체에서 사용하는 오류 코드를 HTTP 상태 코드, 메시지, 발생 조건과 함께 열거한다.

### 5. 인증·보안
인증 방식, 토큰 알고리즘 및 유효 기간, HTTPS 강제 여부, Rate Limiting, CORS 허용 도메인을 정의한다.

### 6. 변경 이력
버전별 변경 내용, 작성자, 승인자를 기록한다.

## 주요 표 템플릿

### API 목록 표

| API ID | 메서드 | URL | 설명 | 인증 |
|--------|--------|-----|------|------|
| {API-모듈-001} | {GET\|POST\|PATCH\|DELETE} | {/api/v1/...} | {기능 설명} | {불필요\|Bearer} |

### Request Body 필드 정의 표

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| {field_name} | {string\|integer\|UUID\|array} | {필수\|선택} | {설명 및 제약 조건} |

### Path / Query Parameters 표

| 파라미터 | 타입 | 설명 |
|---------|------|------|
| {param_name} | {UUID\|string\|integer} | {설명} |

### 오류 코드 정의 표

| 오류 코드 | HTTP 상태 | 메시지 | 발생 조건 |
|---------|---------|--------|---------|
| {ERROR_CODE} | {400\|401\|403\|404\|409\|422\|429\|500\|503} | {사용자 노출 메시지} | {발생 시나리오} |

### 인증·보안 표

| 항목 | 내용 |
|------|------|
| 인증 방식 | {JWT Bearer Token 등} |
| 액세스 토큰 알고리즘 | {HS256 등} |
| 액세스 토큰 유효시간 | {시간 (초)} |
| 리프레시 토큰 유효시간 | {일 (초)} |
| HTTPS 강제 | {적용 여부} |
| Rate Limiting | {정책 설명} |
| CORS | {허용 도메인} |

### 변경 이력 표

| 버전 | 변경일 | 변경 내용 | 작성자 | 승인자 |

| {v0.1} | {YYYY-MM-DD} | {초안 작성} | {이름} | - |
| {v1.0} | {YYYY-MM-DD} | {변경 내용 요약} | {이름} | {이름} |

## 작성 팁

- **API ID 체계를 사전에 확정한다.** `API-{모듈코드}-{순번}` 형식(예: API-AUTH-001)으로 통일하면 화면 설계서, 인터페이스 설계서와 교차 참조가 용이하다.
- **비동기 API는 완료 확인 방식을 명시한다.** 202 Accepted를 반환하는 AI 생성·배치 처리 API는 반드시 WebSocket 또는 Polling 엔드포인트와 함께 기술한다.
- **오류 코드는 도메인 단위로 구분하여 정의한다.** AUTH_, DOC_, PROJECT_ 등 접두사를 붙이면 원인 영역을 즉시 식별할 수 있으며, 프론트엔드의 오류 메시지 처리 로직 작성이 쉬워진다.
