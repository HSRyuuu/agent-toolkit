# 프로그램 사양서 작성 가이드

## 목적
시스템을 구성하는 개별 프로그램(서비스, 배치 등)의 입출력, 처리 로직 흐름, 관련 테이블, 예외 처리를 상세히 정의한다.
개발자가 코드를 작성하기 전에 로직 수준의 설계를 확정하고, 테스트 케이스 도출의 기준이 된다.

## 작성 시점
설계 단계 후반 — API 명세서, DB 설계서, 화면 설계서가 모두 확정된 후 작성한다. 개발 착수 직전 최종 검토 문서로 활용된다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 프로그램사양서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v0.1} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {이름 (역할)} |
| 승인자 | {이름 (역할)} |

---

## 필수 섹션 구조

### 1. 프로그램 목록
시스템을 구성하는 모든 프로그램을 프로그램 ID, 프로그램명, 유형(화면/배치/공통), 담당자, 비고로 요약한다.

### 2. 프로그램 상세 사양
각 프로그램별 하위 절(###)로 구성하며, 다음 항목을 순서대로 기술한다.
- 입출력 정의 표 (구분, 항목, 데이터 타입, 설명)
- 처리 로직 흐름도 (단계별 번호 처리, 분기 조건, 예외 발생 지점 포함)
- 관련 테이블 표 (테이블명, 작업 유형, SQL 개요)
- 예외 처리 표 (예외 상황, 오류 메시지, 처리 방안)

배치 프로그램의 경우 유형(이벤트 기반/스케줄 등)과 트리거 조건을 추가로 명시한다.

### 3. 공통 모듈
여러 프로그램에서 공유하는 공통 모듈(미들웨어, 헬퍼, 데코레이터 등)의 기능과 사용 프로그램을 표로 정의한다.

### 변경 이력
버전별 변경 내용, 작성자, 승인자를 기록한다.

## 작성 팁

- **처리 로직 흐름도에 예외 발생 지점을 명시한다.** 정상 흐름만 작성하면 개발자가 예외 처리를 임의로 구현하게 된다. 각 단계에서 발생 가능한 예외 조건(없음/권한 없음/외부 API 오류 등)과 분기를 흐름도에 포함시킨다.
- **배치 프로그램은 트리거 방식과 멱등성 보장 방법을 명시한다.** 이벤트 기반인지 스케줄 기반인지, 동일 이벤트가 중복 수신될 때 중복 처리를 어떻게 방지하는지(Upsert, 이벤트 ID 중복 체크 등)를 기술한다.
- **프로그램 ID는 API ID, 화면 ID와 연계하여 추적 가능하게 설계한다.** 예: PRG-DOC-001(문서 관리 서비스) ↔ API-DOC-001~005 ↔ SCR-DOC-001~003. 이 연계 관계를 비고 또는 프로그램 설명에 명시하면 변경 영향도 분석이 용이하다.
