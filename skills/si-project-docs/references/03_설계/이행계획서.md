# 이행계획서 작성 가이드

## 목적

기존 시스템에서 신규 시스템으로 데이터와 서비스를 안전하게 이행하기 위한 절차를 정의한다. 이행 대상 범위, 단계별 절차, 데이터 매핑, 검증 방법, 롤백 계획을 사전에 수립하여 이행 중 데이터 손실·서비스 중단 위험을 최소화한다.

## 작성 시점

설계 단계 후반 — DB 설계서 완료 후 작성한다. 이행 리허설 수행 후 실측 결과를 반영하여 갱신하고, 본 이행 전날까지 최종 확정한다.

## 필수 섹션 구조

### 1. 이행 대상 목록

이행할 데이터·서비스의 전체 범위를 표로 정리한다. 이행 ID, 대상 구분(데이터/서비스/설정), 원천 시스템, 신규 시스템, 이행 방식, 우선순위를 포함한다.

### 2. 이행 전략

Big-Bang(일괄 이행) 또는 점진적 이행(단계별 전환) 방식을 결정하고 근거를 기술한다. 이행 중 구 시스템과 신규 시스템의 병행 운영 여부 및 기간도 명시한다.

### 3. 이행 절차

이행을 단계별로 세분화하여 각 단계의 담당자, 소요 시간, 확인 방법을 기재한다. 단계는 번호 순서로 작성하여 이행 당일 체크리스트로 활용할 수 있어야 한다.

- **사전 준비**: 이행 환경 점검, 백업 수행, 서비스 중단 공지
- **데이터 추출**: 원천 시스템에서 데이터 Export
- **데이터 변환**: 매핑 규칙에 따른 변환 및 정제
- **데이터 적재**: 신규 시스템 DB에 Import
- **검증**: 건수 검증, 샘플 검증, 업무 검증
- **서비스 전환**: DNS 변경, 로드밸런서 전환 등
- **모니터링**: 전환 후 안정화 기간 모니터링

### 4. 이행 일정

리허설 이행과 본 이행의 일정을 구분하여 표로 기재한다. 각 단계별 시작·종료 시각, 담당자, 완료 기준을 포함한다.

### 5. 데이터 매핑 표

원천 시스템의 테이블·컬럼을 신규 시스템의 테이블·컬럼으로 매핑한다. 변환 규칙(직접 매핑, 코드 변환, 기본값 설정, 신규 생성 등)과 비고를 함께 기재한다.

### 6. 검증 방법

이행 완료 후 데이터 정합성을 확인하는 방법을 정의한다. 건수 검증(원천 vs 신규), 샘플 검증(임의 N건 비교), 업무 검증(주요 기능 동작 확인)의 세 단계로 구성한다.

### 7. 롤백 계획

이행 실패 또는 검증 오류 발생 시 원래 시스템으로 복구하는 절차를 정의한다. 롤백 결정 기준(임계 조건), 롤백 절차, 소요 시간, 담당자를 명시한다.

### 8. 변경 이력

버전별 변경 내용을 추적한다.

## 작성 팁

- **MIG ID로 추적성을 확보한다.** `MIG-{순번}` 형식(예: MIG-001)으로 이행 항목에 ID를 부여하고, 검증 결과와 롤백 로그에도 동일 ID를 참조하면 이행 후 이슈 추적이 쉬워진다.
- **리허설 이행을 반드시 수행한다.** 본 이행 1~2주 전 동일 절차로 리허설을 수행하고 소요 시간과 오류를 기록한다. 리허설 결과를 반영하여 본 이행 계획을 수정한다.
- **이행 시간대를 서비스 최소 사용 시간으로 설정한다.** 이행 중 서비스 중단이 불가피한 경우 새벽 시간대나 주말을 활용하고, 사전 공지 일정도 이행 일정 표에 포함시킨다.
- **롤백 기준을 정량적으로 정의한다.** "오류가 많으면 롤백"이 아닌 "검증 오류 건수가 전체의 1% 초과 시 롤백"처럼 수치로 명시해야 이행 현장에서 신속하게 결정할 수 있다.
