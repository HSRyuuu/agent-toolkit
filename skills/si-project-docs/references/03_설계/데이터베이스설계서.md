# 데이터베이스 설계서 작성 가이드

## 목적
시스템의 데이터 구조를 논리 모델(ERD)과 물리 모델(테이블 정의서)로 정의한다.
DB 개발자와 백엔드 개발자 간의 스키마 기준을 통일하고, 발주처의 데이터 구조 검토 근거를 제공한다.

## 작성 시점
설계 단계 초중반 — 도메인 모델링이 완료된 시점에 논리 ERD를 먼저 작성하고, 기술 스택(DBMS) 확정 후 물리 모델로 전환한다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 데이터베이스설계서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v0.1} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {이름 (역할)} |
| 승인자 | {이름 (역할)}, {이름 (역할)} |

---

## 필수 섹션 구조

### 1. 데이터 모델링 방향
논리/물리 모델 설계 원칙(정규화 기준, DBMS 버전, 벡터 확장 등), 소프트 삭제 정책, 감사 로그 컬럼 규칙을 간략히 서술한다.

### 2. ERD (논리 모델)
주요 엔티티 간의 관계를 텍스트 다이어그램으로 표현한다. 1:N, N:M 등 카디널리티와 연결 테이블을 명시한다.

### 3. 테이블 정의서
각 테이블별 하위 절(###)로 구성하며, 컬럼명·데이터 타입·NULL 여부·기본값·설명을 표로 정의하고, 인덱스 정의를 항목으로 추가한다.

### 4. 공통 코드 정의서
시스템 전체에서 사용하는 코드 그룹과 코드값(코드명, 설명)을 열거한다.

### 5. 데이터 이행 계획
기존 시스템에서 신규 시스템으로의 데이터 이행 범위, 담당자, 일정, 비고를 정의한다. 신규 시스템의 경우 초기 데이터 적재 계획으로 대체한다.

### 변경 이력
버전별 변경 내용, 작성자, 승인자를 기록한다.

## 주요 표 템플릿

### 테이블 정의 표

| 컬럼명 | 데이터 타입 | NULL | 기본값 | 설명 |
|--------|-----------|------|--------|------|
| {table_id} | UUID | NOT NULL | gen_random_uuid() | PK, 고유 ID |
| {fk_column} | UUID | NOT NULL | - | FK → {참조테이블}.{참조컬럼} |
| {name_column} | VARCHAR({N}) | NOT NULL | - | {컬럼 설명} |
| {text_column} | TEXT | NULL | - | {컬럼 설명} |
| {status_column} | VARCHAR(20) | NOT NULL | '{기본값}' | 상태 코드 ({값1}/{값2}) |
| created_by | UUID | NOT NULL | - | FK → TB_USER.user_id |
| created_at | TIMESTAMPTZ | NOT NULL | NOW() | 생성 일시 |
| updated_at | TIMESTAMPTZ | NOT NULL | NOW() | 수정 일시 |
| deleted_at | TIMESTAMPTZ | NULL | - | 삭제 일시 (소프트 삭제) |

> 인덱스:
> - `idx_{table}_{column}` {UNIQUE} ON ({컬럼}) {WHERE 조건}

### 공통 코드 정의 표

| 코드 그룹 | 코드값 | 코드명 | 설명 |
|---------|--------|--------|------|
| {CODE_GROUP} | {CODE_VALUE} | {코드명} | {설명} |

### 데이터 이행 계획 표

| 작업 | 담당자 | 일정 | 비고 |
|------|--------|------|------|
| {이행/적재 작업명} | {이름} | {YYYY-MM-DD ~ MM-DD} | {처리 방법 등 비고} |

### 변경 이력 표

| 버전 | 변경일 | 변경 내용 | 작성자 | 승인자 |
|------|--------|----------|--------|--------|
| {v0.1} | {YYYY-MM-DD} | {논리 ERD 초안} | {이름} | - |
| {v1.0} | {YYYY-MM-DD} | {물리 모델 확정} | {이름} | {이름} |

## 작성 팁

- **소프트 삭제와 감사 컬럼은 모든 주요 테이블에 일괄 적용한다.** `deleted_at`, `created_at`, `updated_at`, `created_by`, `updated_by`를 공통 규칙으로 정의하면 테이블 정의서 작성이 일관되고 누락을 방지할 수 있다.
- **테이블명과 컬럼명 명명 규칙을 첫 절에 명시한다.** `TB_` 접두사, snake_case 등 규칙을 사전에 정해두면 여러 명이 분담 작성할 때 불일치를 막을 수 있다.
- **N:M 관계는 반드시 연결 테이블로 분리하여 ERD에 표현한다.** 연결 테이블의 컬럼 구성(복합 PK 또는 별도 PK)과 인덱스 전략을 함께 기술해야 쿼리 성능 설계의 근거가 된다.
