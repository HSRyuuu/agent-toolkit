# 인터페이스 설계서 작성 가이드

## 목적
시스템이 연동하는 외부 시스템(LLM API, SSO, 이메일, Git 등)과의 인터페이스 목록, 데이터 흐름, 전문 구조, 오류 처리 전략을 정의한다.
내부 시스템과 외부 시스템 간의 연계 계약을 문서화하여 개발·운영 단계의 혼선을 방지한다.

## 작성 시점
설계 단계 후반 — API 명세서 및 시스템 아키텍처 설계서 확정 후 작성한다. 외부 연동 API의 계약이 확정된 시점에 세부 전문 구조를 기술한다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 인터페이스설계서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v0.1} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {이름 (역할)} |
| 승인자 | {이름 (역할)} |

---

## 필수 섹션 구조

### 1. 인터페이스 목록
시스템이 연동하는 모든 외부 인터페이스를 I/F ID, 인터페이스명, 송신/수신 시스템, 유형(실시간/배치), 연계 방식으로 요약한다.

### 2. 데이터 흐름
주요 인터페이스별로 데이터가 송신 시스템에서 수신 시스템까지 이동하는 단계별 처리 흐름을 텍스트 다이어그램으로 표현한다.

### 3. 전문 구조
각 인터페이스의 요청/응답 전문 포맷(JSON, XML 등)을 코드 블록 예시와 필드 정의 표로 기술한다.

### 4. 연계 방식 상세
각 I/F ID별로 프로토콜, 인증 방식, 타임아웃, 재시도 정책을 표로 정의한다.

### 5. 오류 처리
각 I/F ID별로 발생 가능한 오류 유형, 재처리 방안, 알림 방법을 표로 정의한다.

### 변경 이력
버전별 변경 내용, 작성자, 승인자를 기록한다.

## 작성 팁

- **I/F ID 체계를 도메인 단위로 구분한다.** `IFC-{도메인}-{순번}` 형식(예: IFC-LLM-001, IFC-SSO-001)으로 정의하면 인터페이스 유형별 묶음 관리와 오류 추적이 용이하다.
- **폴백 전략을 오류 처리 표에 명시한다.** 외부 API 장애 시 대체 수단(예: 1차 LLM 실패 시 2차 LLM으로 전환)이 있다면 오류 유형별로 폴백 흐름까지 함께 정의한다.
- **데이터 흐름도에서 민감 정보 처리 단계를 명시한다.** 개인정보나 인증 키가 포함된 전문이 외부로 전달되는 경우, 마스킹 또는 암호화 처리 단계를 흐름도 상에 명확히 표기해야 보안 검토 시 근거가 된다.
