# 시스템 아키텍처 설계서 작성 가이드

## 목적
시스템의 레이어 구조, 기술 스택, 컴포넌트 의존 관계, 배포 환경, 보안 및 고가용성 설계를 정의한다.
아키텍처 의사결정의 근거를 문서화하고, 팀 전체가 동일한 기술 방향 위에서 개발할 수 있도록 기준을 제공한다.

## 작성 시점
설계 단계 초반 — 요구사항 정의서 확정 직후 가장 먼저 작성하며, 다른 설계 문서(DB 설계서, API 명세서 등)의 기술 스택 기준이 된다.
발주처 검토 후 기술 스택 및 배포 구조를 확정하여 v1.0을 완성한다.

## 필수 섹션 구조

### 1. 아키텍처 개요

#### 1.1 설계 원칙
관심사 분리, 확장성, 고가용성, 보안, 관찰 가능성 등 아키텍처 결정의 핵심 원칙과 각 원칙의 적용 내용을 표로 정의한다.

#### 1.2 레이어 구조
Presentation / Application / Domain / Infrastructure 등 계층 구조를 텍스트 다이어그램으로 표현하고, 각 레이어의 역할과 계층 간 통신 방식을 명시한다.

### 2. 기술 스택
Frontend, Backend, DB, 캐시, LLM, 컨테이너, CI/CD, 클라우드, 모니터링 등 영역별 기술·버전·선택 근거를 표로 정의한다.

### 3. 컴포넌트 다이어그램
주요 컴포넌트(모듈, 서비스, 어댑터 등)와 의존 관계를 텍스트 다이어그램으로 표현한다.

### 4. 배포 다이어그램
소스 코드 저장소부터 CI/CD 파이프라인, 컨테이너 이미지 레지스트리, 컴퓨팅 환경, 데이터 스토리지까지의 배포 흐름을 텍스트 다이어그램으로 표현한다.

### 5. 보안 아키텍처

#### 5.1 인증·인가 흐름
로그인 요청부터 JWT 발급, API 요청 시 토큰 검증, RBAC 권한 확인까지의 흐름을 텍스트 다이어그램으로 표현한다.

#### 5.2 API 게이트웨이 구성
Rate Limiting, 요청 크기 제한, CORS, SQL Injection 방어, XSS 방어 등 보안 구성 항목을 표로 정의한다.

### 6. 고가용성 설계
컴퓨팅, DB, 캐시, 스토리지 등 구성 요소별 이중화 방식과 RTO/RPO 목표를 표로 정의한다.

### 변경 이력
버전별 변경 내용, 작성자, 승인자를 기록한다.

## 작성 팁

- **기술 선택 근거를 반드시 기술한다.** 단순히 기술명과 버전만 나열하면 발주처 검토나 향후 기술 교체 시 판단 근거가 없어진다. 팀 역량, 생태계 성숙도, 성능 특성 등 2~3가지 이유를 간략히 명시한다.
- **레이어 다이어그램과 컴포넌트 다이어그램을 분리하여 작성한다.** 레이어 구조는 계층 간 책임 분리를, 컴포넌트 다이어그램은 모듈 간 실제 의존 관계를 보여주며, 두 관점은 목적이 다르다.
- **고가용성 표에 RTO/RPO를 반드시 명시한다.** 장애 복구 목표를 수치로 정의해야 인프라 구성(이중화 여부, 복제 방식)의 수준을 발주처와 합의할 수 있고, 운영 단계의 SLA 기준이 된다.
