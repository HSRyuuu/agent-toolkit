# 화면 설계서 작성 가이드

## 목적
시스템의 모든 화면 목록, 와이어프레임, 입력 필드 정의, 사용자 흐름도, 디자인 가이드를 정의한다.
프론트엔드 개발자, UI/UX 디자이너, 발주처 간의 화면 구성 합의 기준 문서 역할을 한다.

## 작성 시점
설계 단계 — 요구사항 정의서 확정 후 착수하며, API 명세서와 병행하여 작성한다.
발주처 UX 검토 후 디자인 가이드를 확정하여 v1.0을 완성한다.

## 필수 섹션 구조

### 1. 화면 목록
전체 화면을 화면 ID, 화면명, 메뉴 경로, URL, 설명으로 정리한 요약 목록.

### 2. 주요 화면 와이어프레임
핵심 화면을 ASCII 아트 또는 이미지로 표현한다. 레이아웃 구성 요소(헤더, 사이드바, 카드, 테이블 등)의 배치를 명시한다.

### 3. 화면 상세 정의
각 화면별로 입력 요소(컴포넌트 유형, 필수 여부, 유효성 검사 규칙, 설명)를 표로 정의하고, 화면별 유효성 검사 규칙을 항목으로 추가한다.

### 4. 사용자 흐름도
주요 업무 시나리오별로 화면 전환 흐름을 텍스트 다이어그램으로 표현한다.

### 5. 디자인 가이드
색상 팔레트, 타이포그래피, 컴포넌트 규격(높이·패딩·테두리 반경)을 표로 정의한다.

### 6. 반응형 정책
지원 해상도 구간별 레이아웃 정책을 정의한다.

### 변경 이력
버전별 변경 내용, 작성자, 승인자를 기록한다.

## 작성 팁

- **화면 ID 체계를 메뉴 구조와 일치시킨다.** `SCR-{모듈코드}-{순번}` 형식(예: SCR-DOC-001)으로 정의하면 API 명세서의 URL 경로와 1:1 대응 추적이 쉬워진다.
- **유효성 검사 규칙은 API Request Body 제약 조건과 반드시 일치시킨다.** 화면 단 검증과 서버 단 검증의 불일치는 QA 단계에서 혼란을 유발하므로, 작성 시 API 명세서와 교차 확인한다.
- **사용자 흐름도는 정상 흐름과 예외 흐름을 함께 표현한다.** 승인/반려, 오류 팝업, 로딩 상태 등 분기점을 명시해야 프론트엔드 개발 누락을 방지할 수 있다.
