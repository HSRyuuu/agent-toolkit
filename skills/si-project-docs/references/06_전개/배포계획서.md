# 배포계획서 작성 가이드

## 목적
운영 환경 배포 전 전체 배포 절차, 일정, 롤백 계획을 사전에 문서화하여 이해관계자 간 합의를 확보하고 배포 리스크를 최소화한다.

## 작성 시점
전개(배포) 단계 착수 직전, 스테이징 리허설 완료 후 최종 배포 약 1주일 전에 작성하여 발주처 승인을 받는다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 배포계획서 |
| 프로젝트명 | {프로젝트명} |
| 배포 버전 | {v0.0.0} |
| 배포 예정일 | {YYYY-MM-DD (요일) HH:MM ~ HH:MM} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {작성자명 (역할)}, {작성자명 (역할)} |
| 승인자 | {승인자명 (발주처)} |

## 필수 섹션 구조

### 1. 배포 목표
배포 버전, 배포 대상 환경(URL), 배포 내용 요약, 배포 시간대, 예상 서비스 영향 범위를 5행 표로 정리한다.

### 2. 배포 방식
채택한 배포 전략(Blue-Green, Rolling, Canary 등)을 명시하고, 아키텍처 흐름을 텍스트 다이어그램으로 표현한다. 해당 전략을 선택한 근거(무중단, 롤백 용이성 등)를 함께 기술한다.

### 3. 배포 일정
D-N일 단위 사전 준비부터 배포 당일 시간대별 작업까지 타임라인 표로 작성한다. 각 작업마다 담당자를 지정한다.

### 4. 사전 점검 사항
배포 전 완료해야 할 체크리스트를 순번, 점검 항목, 담당자, 완료 여부(☐) 형식의 표로 작성한다.

### 5. 배포 절차 (단계별)
Step 단위로 실제 실행 명령어와 확인 방법을 포함하여 작성한다. 주요 Step 예시: DB 마이그레이션 → 서비스 배포 → 헬스체크/스모크 테스트 → 트래픽 전환.

### 6. 롤백 계획
롤백 발동 조건, 결정권자, 롤백 방법, 소요 시간, DB 롤백 여부를 표로 정리하고, 실제 롤백 명령어를 코드 블록으로 제시한다.

### 7. 이해관계자 공지
공지 시점(배포 전 D-N일, 배포 완료 시, 장애 발생 시), 대상, 내용, 방법(이메일/인앱/전화)을 표로 정리한다.

### 변경 이력
버전, 변경일, 변경 내용, 작성자, 승인자를 표로 관리한다.

## 주요 표 템플릿

**배포 일정 표**

| 시간 | 작업 내용 | 담당자 |
|------|---------|--------|
| D-{N} ({날짜}) | {사전 준비 작업} | {담당자} |
| {날짜} {시각} | **배포 시작** — {첫 번째 작업} | {담당자} |
| {날짜} {시각} | {중간 작업} | {담당자} |
| {날짜} {시각} | **배포 완료** 선언 및 이해관계자 공지 | {담당자} |

**사전 점검 체크리스트**

| 순번 | 점검 항목 | 담당자 | 완료 여부 |
|------|---------|--------|---------|
| 1 | {점검 항목} | {담당자} | ☐ |
| 2 | {점검 항목} | {담당자} | ☐ |

**롤백 계획 표**

| 항목 | 내용 |
|------|------|
| 롤백 조건 | {오류율 N% 초과, Critical 장애 발생 등} |
| 롤백 결정권자 | {PM명} |
| 롤백 방법 | {구체적 방법} |
| 롤백 소요 시간 | 약 {N}분 이내 |
| DB 롤백 | {필요 시 조건 및 방법} |

**이해관계자 공지 계획**

| 공지 시점 | 대상 | 내용 | 방법 |
|---------|------|------|------|
| D-{N} | {대상} | {내용} | {이메일/인앱/전화} |
| 배포 완료 시 | {대상} | {내용} | {방법} |
| 장애 발생 시 | {발주처·핵심 이해관계자} | 상황 보고 및 예상 복구 시간 | 긴급 전화·메신저 |

## 작성 팁

- 배포 일정 표의 각 작업에는 반드시 담당자를 명시하고, 배포 시작·완료 시점은 굵게 강조하여 시각적으로 구분한다.
- 롤백 명령어는 실제 실행 가능한 형태로 코드 블록에 작성하되, 민감한 ARN/키 값은 환경변수(`${VAR_NAME}`) 형식으로 대체한다.
- 스테이징 환경 리허설 완료 후 발견된 문제점과 개선사항을 "배포 방식" 또는 "사전 점검 사항"에 반드시 반영한다.
