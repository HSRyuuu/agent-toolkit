# 비상대책방안 작성 가이드

## 목적
서비스 운영 중 발생할 수 있는 장애 유형을 사전에 정의하고, 유형별 탐지·조치·복구·보고 절차를 문서화하여 장애 발생 시 신속하고 일관된 대응을 가능하게 한다.

## 작성 시점
전개 단계 착수 전(오픈 계획서와 함께) 작성하여 발주처 승인을 받는다. 시스템 구조 변경 시 내용을 갱신한다.

## 필수 섹션 구조

### 1. 장애 유형 분류
장애 등급(P1~P4)별 정의와 구체적 예시를 표로 작성한다. P1(전체 서비스 불가)부터 P4(경미한 UI·성능 이슈)까지 등급 기준을 명확히 한다.

### 2. 대응 절차

#### 2.1 장애 탐지 → 분석 → 조치 → 복구
탐지 → 1차 확인 → 등급 분류 → 원인 분석 → 조치 → 복구 확인 → 보고 흐름을 텍스트 다이어그램으로 표현한다. P1/P2와 P3/P4의 대응 경로를 분기하여 표현한다.

### 3. 장애 시나리오별 대응 절차
시스템 구성 요소별(웹서버, DB, 외부 API, 네트워크 등) 장애 시나리오를 각각 소제목으로 나누어, 탐지·1차 조치·2차 조치·확인 단계와 담당자·소요 시간을 표로 작성한다.

### 4. 복구 목표
장애 등급별 RTO(복구 목표 시간)와 RPO(데이터 복구 목표 시점)를 표로 정리한다.

### 5. 롤백 절차

#### 5.1 롤백 결정 기준
롤백을 실행해야 하는 조건(오류율 임계값, P1/P2 장애 발생, PM 판단 기준)을 불릿으로 명시한다.

#### 5.2 롤백 실행 절차
실제 실행 명령어를 코드 블록으로 제시하고, 각 단계에 주석으로 목적을 설명한다.

### 6. 연락·보고 체계
장애 등급별 즉시 보고 대상, 30분 후 업데이트 보고, 복구 후 사후 보고 의무를 표로 정리한다.

### 변경 이력
버전, 변경일, 변경 내용, 작성자, 승인자를 표로 관리한다.

## 작성 팁

- 장애 시나리오는 시스템 아키텍처의 주요 구성 요소(웹서버, DB, 캐시, 외부 API, 네트워크) 각각에 대해 최소 하나씩 작성한다.
- 롤백 명령어는 실제 배포 환경의 ARN·클러스터명 등 민감 정보를 환경변수(`${VAR_NAME}`)로 처리하여 문서에 기재한다.
- 복구 목표(RTO/RPO)는 SLA와 수치를 일치시키고, 달성 가능한 현실적인 값으로 설정하여 발주처와 합의한다.
