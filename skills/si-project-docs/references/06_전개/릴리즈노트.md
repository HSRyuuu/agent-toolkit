# 릴리즈노트 작성 가이드

## 목적
새로운 버전의 소프트웨어를 배포할 때 신규 기능, 변경 사항, 버그 수정, 알려진 문제, 업그레이드 방법을 사용자와 이해관계자에게 명확히 전달한다.

## 작성 시점
배포(릴리즈) 직전, 배포 계획서 승인 이후에 작성하여 배포 완료와 함께 공개한다. 베타·스테이징 테스트 결과가 확정된 시점에 작성을 시작한다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 릴리즈노트 |
| 프로젝트명 | {프로젝트명} |
| 릴리즈 버전 | {v0.0.0} |
| 릴리즈 일자 | {YYYY-MM-DD} |
| 작성자 | {작성자명 (역할)} |

## 필수 섹션 구조

### {시스템명} {버전} 릴리즈 노트 (인트로 요약)
릴리즈의 성격과 포함 내용을 1~2문장으로 요약한 인트로 블록쿼트(`>`)를 작성한다.

### 1. 신규 기능
기능 그룹별로 소제목(`###`)을 나누어 구체적인 기능 목록을 불릿으로 기술한다. 사용자가 실제로 체감할 수 있는 기능 중심으로 작성한다.

### 2. 변경 사항
기존 동작 방식이 변경된 항목을 기술한다. 최초 릴리즈의 경우 "기존 버전 대비 변경 사항 없음"으로 명시한다.

### 3. 버그 수정
베타·스테이징 단계에서 수정된 주요 결함을 결함 ID와 내용으로 표로 정리한다.

### 4. 알려진 문제 (Known Issues)
현재 버전에 남아 있는 미해결 이슈를 이슈 ID, 내용, 회피 방안, 예상 수정 버전으로 표로 정리한다.

### 5. 업그레이드 가이드
기존 버전에서 이 버전으로 전환하는 방법, 신규 설치 방법, 지원 브라우저/OS 환경, 최소 해상도 등 접속 요건을 기술한다.

### 6. 다음 릴리즈 예고 ({다음 버전} - {예정 시기})
다음 버전에서 제공될 기능 목록을 체크박스(`- [ ]`) 형식으로 나열한다.

### 문의 및 지원
헬프데스크 이메일, 긴급 장애 연락처, 메신저 채널 등을 표로 제공한다.

## 주요 표 템플릿

**버그 수정 표**

| 결함 ID | 내용 |
|---------|------|
| DEF-{NNN} | {증상} → {수정 내용} |

**알려진 문제 표**

| 이슈 | 내용 | 회피 방안 | 예상 수정 버전 |
|------|------|---------|------------|
| IS-{NNN} | {이슈 내용} | {임시 회피 방법} | {v0.0.0 (YYYY-MM 예정)} |

**문의 및 지원 표**

| 채널 | 내용 |
|------|------|
| 서비스 헬프데스크 | {이메일 주소} |
| 긴급 장애 | {연락처} ({담당자명}) |
| 메신저 채널 | {채널명} |

## 작성 팁

- 신규 기능은 기능 그룹(도메인)별로 소제목을 나누어 작성하면 독자가 관심 기능을 빠르게 찾을 수 있다.
- "알려진 문제"는 숨기지 않고 반드시 명시하며, 회피 방안과 수정 예정 버전을 함께 제공하여 사용자 신뢰를 유지한다.
- 업그레이드 가이드는 비기술 사용자(PM, BA)도 따라할 수 있도록 단계별 번호 목록으로 작성한다.
