# 백업및복구계획서 작성 가이드

## 목적
시스템 운영 중 데이터 손실이나 장애 발생 시 신속하게 복구할 수 있도록 백업 대상, 주기, 방식, 보관 기간, 복구 절차, 복구 테스트 계획을 사전에 정의한다.

## 작성 시점
전개 단계 완료 직후(공식 오픈 시점)에 작성하여 운영팀에 인계한다. 인프라 구성 변경 시 갱신한다.

## 필수 섹션 구조

### 1. 백업 대상
백업 대상 데이터를 대상(시스템 구성요소), 데이터 내용, 예상 크기, 중요도(최상/상/중/하)로 표로 정리한다. 중요도가 높은 항목(DB, 시크릿)을 상단에 배치한다.

### 2. 백업 주기
데이터 유형별로 소제목을 나누어(DB, 파일 저장소, 설정 및 코드) 백업 유형, 주기, 방식, 보관 기간, 담당을 표로 작성한다.

#### 2.1 데이터베이스
자동 스냅샷, 수동 스냅샷, Point-in-Time 복구, 월간 아카이브 등 유형별 주기와 보관 기간을 분리하여 기술한다.

#### 2.2 파일 저장소
버저닝 활성화, Cross-Region 복제 등 파일 스토리지의 백업 방식을 기술한다.

#### 2.3 설정 및 코드
배포 설정, 소스 코드, 시크릿 관리 설정의 백업 방식과 주기를 기술한다.

### 3. 백업 방식
자동 백업 설정 확인 명령어, 수동 스냅샷 생성 명령어를 코드 블록으로 제공한다. 백업 설정의 주요 파라미터(보관 기간, 백업 윈도우, Multi-AZ 여부 등)를 확인하는 명령어 출력 예시도 함께 포함한다.

### 4. 보관 기간
데이터 유형별 보관 기간과 그 근거(비용·복구 필요성 균형, 감사 요건 등)를 표로 정리한다.

### 5. 복구 절차
복구 유형별로 소제목을 나누어 실제 실행 명령어와 대기 명령어, 복구 결과 확인 명령어를 코드 블록으로 제공한다.

#### 5.1 DB 특정 시점 복구 (Point-in-Time)
복구 시점 결정 → 새 인스턴스 생성 → 복구 완료 대기 → 엔드포인트 확인 순서로 명령어를 제시한다.

#### 5.2 스냅샷 복구
사용 가능한 스냅샷 목록 확인 → 스냅샷으로 인스턴스 복원 순서로 명령어를 제시한다.

#### 5.3 파일 스토리지 이전 버전 복구
특정 파일의 버전 목록 조회 → 특정 버전으로 복구 순서로 명령어를 제시한다.

### 6. 복구 테스트 계획
테스트 항목, 주기, 담당자, 테스트 방법을 표로 정리한다. DR(재해복구) 전환 훈련도 포함한다.

### 변경 이력
버전, 변경일, 변경 내용, 작성자, 승인자를 표로 관리한다.

## 작성 팁

- DB 백업 설정 확인 명령어는 보관 기간(`BackupRetentionPeriod`), 백업 윈도우(`PreferredBackupWindow`), Multi-AZ 여부를 한 번에 확인할 수 있도록 `--query` 파라미터를 활용한다.
- Point-in-Time 복구 시 새로운 인스턴스 이름(`-restored` 접미사 등)으로 생성하고 원본 인스턴스와 병행 확인 후 전환하는 절차를 명확히 기술한다.
- 복구 테스트 주기와 방법을 명시하고 실제 테스트 결과를 변경 이력 또는 별도 테스트 보고서로 관리하는 것을 권장한다.
