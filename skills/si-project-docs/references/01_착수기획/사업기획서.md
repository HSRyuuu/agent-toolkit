# 사업기획서 작성 가이드

## 목적

프로젝트 추진의 배경·목적·범위·예산·일정·리스크를 사업 수준에서 종합 정리하여, 발주처와 수행사 간 사업 방향을 공식적으로 합의하기 위한 문서다.

## 작성 시점

프로젝트 착수 직전 또는 착수 직후 초기 기획 단계에서 작성한다. 발주처 승인 후 프로젝트헌장 작성의 근거 자료로 활용된다.

## 필수 섹션 구조

### 1. 프로젝트 배경 및 목적
- **1.1 추진 배경**: 현재 문제 상황과 수치 근거를 포함한 배경 서술. 주요 문제점을 불릿으로 열거.
- **1.2 추진 필요성**: 이 사업을 추진해야 하는 논거를 1~2문단으로 기술.

### 2. 추진 목표
- **2.1 정량 목표**: 목표 항목·현행 수치·목표 수치·KPI를 표로 정리.
- **2.2 정성 목표**: 정량화하기 어려운 방향성 목표를 불릿으로 열거.

### 3. 사업 범위
- **3.1 기능 범위 (In-Scope)**: 구분·기능을 표로 정리.
- **3.2 제외 범위 (Out-of-Scope)**: 명시적으로 포함하지 않는 항목을 불릿으로 열거.

### 4. 기대 효과
효과 항목·내용·기대 수치를 표로 정리.

### 5. 추진 전략
- **5.1 개발 방법론**: 채택 방법론(Agile/Waterfall 등)과 Sprint/단계 구성 기술.
- **5.2 수행 조직**: 역할·인원·담당 업무를 표로 정리.

### 6. 개략 일정
단계·기간·마일스톤을 표로 정리.

### 7. 개략 예산
구분·금액·비고를 표로 정리. 합계 행을 굵게 표시.

### 8. 리스크 요약
리스크·발생 가능성·영향도·사전 대응 방향을 표로 정리.

### 변경 이력
버전·변경일·변경 내용·작성자·승인자를 표로 관리.

## 작성 팁

- **수치 기반 작성**: 정량 목표와 기대 효과는 반드시 현행 수치와 목표 수치를 함께 기입한다. 근거 없는 수치는 검토 시 신뢰도를 낮춘다.
- **범위 경계 명확화**: In-Scope와 Out-of-Scope를 모두 기술하여 향후 범위 분쟁을 사전에 방지한다. 모호한 항목은 별도 주석으로 판단 기준을 명시한다.
- **리스크와 일정 연동**: 리스크 요약의 대응 방향이 개략 일정의 여유 기간 또는 마일스톤과 연결되도록 작성하면 일관성이 높아진다.
