# 테스트케이스 작성 가이드

## 목적

시스템 전 기능을 빠짐없이 검증하기 위한 테스트 케이스를 사전에 정의하고, 실행 결과(Pass/Fail)와 발견된 결함을 기록하여 테스트 활동의 추적성을 확보한다.

## 작성 시점

통합 테스트 또는 시스템 테스트 착수 직전에 초안을 작성한다. 테스트 수행 결과가 나오면 실제 결과와 판정을 기입하여 갱신하고, 결함 발견 시 버전을 올려 결함 현황을 반영한다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 테스트케이스 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v1.0} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {이름 (역할)} |
| 승인자 | {이름 (역할)} |

## 필수 섹션 구조

### 1. 테스트 케이스 목록 요약
기능 구분별 테스트 케이스 수, Pass 수, Fail 수, 미수행 수를 집계 표로 제공한다. 마지막 행에 합계를 굵게 표시한다. 이 표를 보고 전체 테스트 현황을 한눈에 파악할 수 있어야 한다.

### 2. {기능 구분별} 테스트 케이스
기능 구분별로 소절(`## 2`, `## 3`, ...)을 만들어 테스트 케이스 표를 작성한다. 기능 구분은 테스트 계획서의 In-Scope 범위와 일치시킨다. 각 테이블의 컬럼은 TC ID, 테스트 항목, 요구사항 ID, 선행 조건, 입력 데이터, 실행 절차, 예상 결과, 실제 결과, 판정을 포함한다. Fail 판정인 경우 실제 결과란에 결함 내용을 간략히 기재하고 **굵게** 표시하며, 판정도 **Fail**로 굵게 표기한다.

### {N}. 결함 현황
테스트 케이스 수행 중 발견된 모든 결함을 표로 정리한다. 결함 ID, TC ID, 결함 내용, 심각도(Critical/High/Medium/Low 또는 상/중/하), 조치 상태, 담당자를 포함한다. 결함이 없을 경우 "발견된 결함 없음"으로 명시한다.

### 변경 이력
버전별 변경 내용을 추적한다.

## 작성 팁

- **TC ID와 요구사항 ID 연계**: TC ID는 `TC-{기능약어}-{순번}` 형식으로 통일하고, 요구사항 ID 컬럼에 요구사항정의서의 req_code를 반드시 기재한다. 이렇게 해야 RTM(추적성 매트릭스)에서 요구사항-테스트 케이스 간 추적이 가능하다.
- **실행 절차는 번호 목록으로**: 실행 절차는 "1. ~, 2. ~" 형식의 번호 목록으로 작성하여 재현 가능성을 높인다. 절차가 모호하면 다른 사람이 같은 케이스를 재실행했을 때 다른 결과가 나올 수 있다.
- **Fail 케이스는 즉시 결함 등록**: 판정이 Fail인 케이스는 결함 현황 표에 즉시 결함 ID를 부여하고 조치 담당자를 지정한다. Fail 케이스와 결함 ID 간의 1:1 추적성을 유지한다.
