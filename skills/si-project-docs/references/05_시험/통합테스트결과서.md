# 통합테스트결과서 작성 가이드

## 목적

시스템을 구성하는 각 모듈·서비스 및 외부 연계 시스템이 통합된 환경에서 업무 시나리오 단위로 정상 동작하는지 검증하고, 그 결과를 기록하여 시스템 테스트 및 인수 테스트 진행의 근거로 삼는다.

## 작성 시점

통합 테스트 완료 후 작성한다. 테스트 기간 중 발견된 결함의 조치 및 재검증까지 완료된 시점에 최종본을 작성하여 PM 및 발주처의 승인을 받는다.

## 메타데이터 헤더

| 항목 | 내용 |
|------|------|
| 문서명 | 통합테스트결과서 |
| 프로젝트명 | {프로젝트명} |
| 버전 | {v1.0} |
| 테스트 기간 | {YYYY-MM-DD} ~ {YYYY-MM-DD} |
| 작성일 | {YYYY-MM-DD} |
| 작성자 | {이름 (역할)} |
| 승인자 | {이름 (역할)}, {이름 (역할)} |

## 필수 섹션 구조

### 1. 테스트 시나리오
업무 흐름 기반의 통합 테스트 시나리오 전체를 표로 정리한다. 시나리오 ID, 시나리오명, 업무 흐름 설명(단계별 화살표 흐름으로 표현), 수행 결과(Pass/Fail)를 포함한다. 시나리오는 실제 사용자가 수행하는 end-to-end 흐름을 기준으로 구성한다.

### 2. 연계 시스템 검증
외부 시스템과의 인터페이스를 검증한 결과를 표로 정리한다. 연계 시스템명, 인터페이스 ID(설계 명세서의 IFC ID와 연계), 검증 내용, 결과(Pass/Fail)를 포함한다.

### 3. 시나리오 수행 결과 요약
전체 시나리오의 Pass/Fail 건수와 Pass율을 집계 표로 제공한다.

### 4. 결함 처리 현황
통합 테스트 중 발견된 결함 전체를 표로 정리한다. 결함 ID, 발견된 시나리오 ID, 결함 내용, 심각도, 조치일, 재검증 결과(Pass 시 조치 방법 요약 포함)를 포함한다. 모든 결함이 조치 완료된 경우 표 하단에 "전체 결함 N건 모두 조치 완료"를 굵게 기재한다.

### 5. 미해결 결함
조치가 완료되지 않은 결함이 있을 경우 별도 표로 정리하고, 조치 예정일과 담당자를 명시한다. 미해결 결함이 없는 경우 "현재 미해결 결함 없음"으로 명시한다.

### 변경 이력
버전별 변경 내용을 추적한다.

## 작성 팁

- **시나리오는 업무 관점에서 구성**: 시나리오는 기술적 API 호출이 아니라 "사용자가 무엇을 하는가"를 기준으로 작성한다. 업무 흐름 설명은 `단계1 → 단계2 → 단계3`처럼 화살표로 연결하면 흐름이 명확하게 전달된다.
- **연계 시스템 검증은 IFC ID와 연계**: 연계 시스템 검증 표의 인터페이스 ID는 인터페이스정의서의 IFC ID와 동일한 코드를 사용한다. 이를 통해 설계 - 구현 - 테스트 간 추적성이 확보된다.
- **재검증 결과에 조치 방법 요약 포함**: 결함 처리 현황의 재검증 결과 컬럼에는 단순히 "Pass"만 쓰지 않고, 어떤 방법으로 해결했는지(예: "비동기 큐 처리 방식 개선 후 평균 15초 이내 발송")를 한 줄로 요약하면 이후 유지보수 시 참고 자료가 된다.
