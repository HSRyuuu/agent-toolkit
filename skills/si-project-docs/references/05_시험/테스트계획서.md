# 테스트계획서 작성 가이드

## 목적

시험 단계에서 수행할 테스트 전체를 사전에 계획하고, 테스트 유형·범위·환경·역할·일정·완료 기준·리스크를 이해관계자와 합의하여 테스트 활동의 기준 문서로 삼는다.

## 작성 시점

구축 단계가 일정 수준 진행된 후, 시험 단계 시작 전(통합 테스트 착수 약 2~4주 전)에 작성하여 발주처 및 PM의 승인을 받는다. 일정 변경 등이 발생하면 버전을 올려 갱신한다.

## 필수 섹션 구조

### 1. 테스트 목적·범위
- **1.1 테스트 목적**: 이 테스트를 통해 무엇을 검증하고자 하는지 3~5줄로 기술한다. 기능 요구사항 충족 확인, 비기능 요구사항(성능·보안·가용성) 충족 확인, 운영 배포 전 결함 사전 제거 등을 포함한다.
- **1.2 테스트 범위**: In-Scope(포함 대상)와 Out-of-Scope(제외 대상)를 명확히 구분하여 표로 작성한다. In-Scope 표는 테스트 유형과 대상을 나열하고, Out-of-Scope는 테스트하지 않는 이유와 함께 항목을 명시한다.

### 2. 테스트 유형
수행할 테스트 유형 전체를 표 한 장으로 정리한다. 유형명, 목적, 수행 주체(개발자/QA/외부 업체), 사용 도구를 포함한다.

### 3. 테스트 환경
테스트를 수행하는 환경의 구성을 표로 정리한다. URL, 서버 사양, DB, 외부 연동 시스템, 테스트 데이터 규모, 지원 브라우저 등을 포함한다.

### 4. 역할 및 책임
테스트에 관여하는 모든 역할을 표로 정리한다. 역할명, 담당자, 구체적인 책임 범위를 포함한다.

### 5. 일정
테스트 유형별 시작일, 종료일, 기간을 표로 정리한다. 결함 수정 기간과 최종 완료 확인 일정도 포함한다.

### 6. 완료 기준
테스트를 완료로 판정하는 정량적 기준을 표로 명시한다. 테스트 케이스 실행율, Pass율, 심각도별 미해결 결함 허용 건수, 성능·보안 기준 등을 포함한다.

### 7. 리스크
테스트 활동에 영향을 줄 수 있는 리스크를 표로 정리한다. 리스크 내용, 발생 가능성(상/중/하), 영향도(상/중/하), 대응 방안을 포함한다.

### 변경 이력
버전별 변경 내용을 추적한다.

## 작성 팁

- **Out-of-Scope 명시의 중요성**: 테스트하지 않는 항목을 명확히 적어두지 않으면, 나중에 "왜 이건 테스트 안 했느냐"는 분쟁이 발생한다. 제외 이유(외부 서비스 자체, 인프라 자체, 계약 범위 외 등)를 함께 기재한다.
- **완료 기준은 수치로**: "충분히 테스트했다"는 주관적 판단을 막기 위해 완료 기준을 반드시 정량적 수치로 표현하고, PM·발주처와 사전에 합의하여 문서에 반영한다.
- **리스크는 실행 가능한 대응 방안과 함께**: 리스크를 나열하는 데 그치지 않고, 실제로 실행할 수 있는 구체적인 대응 방안(Mock 서버 구성, 일정 버퍼 확보, 업체 일정 확약서 등)을 함께 기재한다.
