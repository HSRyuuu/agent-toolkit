# 보안취약점점검결과서 작성 가이드

## 목적

운영 배포 전 시스템의 보안 취약점을 자동화 도구와 수동 침투 테스트를 통해 점검하고, 발견된 취약점의 조치 결과를 기록하여 보안 요구사항 충족의 근거로 삼는다.

## 작성 시점

보안 테스트 기간(테스트 계획서에 명시된 보안 테스트 일정)에 수행하고, 발견된 취약점에 대한 조치 및 재점검 완료 후 최종본을 작성한다. 일반적으로 외부 보안 전문 업체가 점검을 수행하고 결과를 제출한다.

## 필수 섹션 구조

### 1. 점검 도구·방법
점검에 사용한 기준, 도구, 방법론을 표 형식으로 정리한다. 점검 기준(OWASP Top 10, 행안부 가이드 등), 자동화 도구(DAST, SAST), 수동 점검 방법, 점검 환경(URL)을 포함한다.

### 2. 취약점 목록
발견된 모든 취약점을 표로 정리한다. 취약점 ID, 취약점 유형, OWASP 분류, 위험도(Critical/High/Medium/Low), 발생 위치(URL 또는 컴포넌트), 조치 결과를 포함한다. 조치 완료된 항목의 위험도는 취소선(`~~High~~`)으로 표시하여 원래 위험도를 보존한다.

### 3. 취약점 상세 및 조치 결과
심각도가 높거나(High/Critical) 설명이 필요한 취약점을 소절(`### {취약점 ID}: {취약점명}`)로 상세히 기술한다. 각 소절에는 발생 위치, 취약점 내용(서술), 취약 코드 또는 설정(코드 블록), 조치 내용 및 수정 코드(코드 블록), 재점검 결과(Pass/Fail과 날짜)를 포함한다.

### 4. 조치 결과 요약
위험도별 발견 건수, 조치 완료 건수, 미조치 건수, 조치율을 집계 표로 제공한다. 마지막 행에 합계를 굵게 표시한다.

### 5. 미조치 취약점 현황
미조치 취약점이 있을 경우 별도 표로 정리하고 조치 예정일과 담당자를 명시한다. 미조치 취약점이 없을 경우 "미조치 취약점 없음 — 발견된 N건 전체 조치 완료"로 명시한다.

### 6. 보안 점검 의견
점검 기관 담당자의 종합 의견(발견 취약점 개요, 조치 평가, 재점검 결과, 운영 배포 적합성 여부)을 자유 서술 형식으로 기재한다. 운영 배포 후 지속적인 보안 관리를 위한 권고 사항(정기 스캔, 의존성 관리, 침투 테스트 주기 등)을 번호 목록으로 포함한다.

### 변경 이력
버전별 변경 내용을 추적한다.

## 작성 팁

- **위험도별 조치 우선순위 명확화**: Critical·High 취약점은 운영 배포 전 반드시 조치 완료해야 한다. Medium 이하는 발주처와 합의하여 운영 후 조치 일정을 수립할 수 있다. 이 기준을 문서 첫머리(또는 테스트 계획서 완료 기준)에 명시한다.
- **취약 코드와 수정 코드를 나란히**: 취약점 상세 소절에서 취약한 코드(Before)와 수정된 코드(After)를 코드 블록으로 나란히 제시하면, 개발자가 유사 패턴을 코드베이스에서 찾아 일괄 수정하는 데 도움이 된다.
- **재점검 날짜 반드시 기재**: 조치 완료 후 반드시 재점검을 수행하고, 재점검 결과에 날짜를 기재한다. 조치했다는 주장만으로는 보안 점검 완료의 근거가 되지 않는다.
