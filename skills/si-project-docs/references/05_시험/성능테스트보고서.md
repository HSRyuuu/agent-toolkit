# 성능테스트보고서 작성 가이드

## 목적

시스템이 요구사항에서 정의한 응답시간·처리량(TPS)·동시 접속자·서버 리소스 사용률 목표를 달성하는지 부하 테스트를 통해 검증하고, 병목 원인 분석 및 개선 조치 결과를 기록한다.

## 작성 시점

시스템 테스트가 안정화된 이후, 운영 배포 전(테스트 계획서의 성능 테스트 기간)에 수행하고 작성한다. 병목 발견 시 튜닝 후 재측정 결과까지 포함하여 최종본을 작성한다.

## 필수 섹션 구조

### 1. 성능 목표치
비기능 요구사항에서 정의한 성능 목표를 표로 정리한다. 항목(응답시간·TPS·동시 접속자·CPU·메모리 등), 목표 기준값, 측정 기준 또는 비고를 포함한다.

### 2. 테스트 도구
성능 테스트에 사용한 도구 전체를 표로 정리한다. 도구명, 버전, 용도(부하 생성/모니터링/분석 등)를 포함한다.

### 3. 시나리오 구성
- **3.1 테스트 시나리오**: 수행한 부하 시나리오(기준 테스트, 램프업, 최대 부하, 내구성, 스파이크 등)를 표로 정리한다. 시나리오명, 부하 설명, 동시 사용자 수, 지속 시간을 포함한다.
- **3.2 테스트 API 구성**: 시나리오 내 호출 API별 가중치(비율)와 비고를 표로 정리한다. 실제 사용 패턴을 반영한 가중치를 적용한다.

### 4. 측정 결과
- **4.1 일반 API 성능**: 최대 부하 시나리오 기준으로 API별 평균 응답시간, P95 응답시간, 최대 응답시간, TPS, 오류율을 표로 정리한다. 마지막 행에 전체 합산 TPS와 오류율을 표시한다. 표 하단에 목표 대비 평가 결과를 항목별로 기재한다.
- **4.2 {특수 기능} 성능**: 응답 시간이 긴 특수 기능(예: AI 생성, 대용량 파일 처리 등)의 성능을 별도 표로 정리한다. 항목, 결과, 목표, 달성 여부를 포함한다.
- **4.3 서버 리소스 사용률**: 최대 부하 시 컴포넌트별 CPU, 메모리 사용률을 표로 정리한다. 목표 초과 항목은 비고란에 명시한다.

### 5. 병목 분석
측정 결과에서 목표를 초과하거나 주의가 필요한 병목 지점을 표로 정리한다. 병목 위치, 원인, 심각도를 포함한다.

### 6. 개선 조치 및 재측정 결과
- **6.1 조치 내용**: 병목별 조치 내용과 조치일을 표로 정리한다.
- **6.2 튜닝 후 재측정 결과**: 주요 지표에 대해 튜닝 전·후 수치, 목표, 달성 여부를 비교 표로 정리한다. 마지막에 최종 평가 한 줄("최종 평가: 모든 성능 목표 달성" 등)을 굵게 표시한다.

### 변경 이력
버전별 변경 내용을 추적한다.

## 작성 팁

- **P95 기준 사용**: 응답시간은 평균값보다 P95(상위 5% 제외 최댓값)를 기준 지표로 사용한다. 평균은 극단값에 의해 왜곡될 수 있으므로, 목표 기준도 P95로 설정하고 측정 결과도 P95 열을 반드시 포함한다.
- **튜닝 전후 비교 표 필수**: 병목이 발견되어 튜닝을 수행한 경우, 튜닝 전후 수치를 동일한 표에 나란히 기재하여 개선 효과를 명확히 보여준다. 수치 비교 없이 "개선되었음"이라는 서술만으로는 근거가 부족하다.
- **서버 리소스는 컴포넌트별 분리**: CPU·메모리 사용률은 전체 시스템 평균이 아닌 컴포넌트(Frontend, Backend API, DB, Cache 등)별로 분리하여 기록한다. 이렇게 해야 어느 컴포넌트에서 스케일업·스케일아웃이 필요한지 특정할 수 있다.
