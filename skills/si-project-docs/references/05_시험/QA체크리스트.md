# QA체크리스트 작성 가이드

## 목적

배포 전 핵심 기능의 정상 동작을 빠르게 검증하는 경량 문서이다. 테스트 계획서·테스트 케이스·통합 테스트 결과서를 1종으로 통합한 간소화 버전으로, 소규모 프로젝트에서 Tier 1 필수 문서로 활용한다. 개발자가 아닌 기획자 또는 다른 팀원이 수행하여 개발자 편향을 방지한다.

> **적용 기준:** 소규모 프로젝트에서는 이 문서 1종이 테스트 계획서 + 테스트 케이스 + 통합 테스트 결과서를 대체한다. 중규모 이상에서는 별도의 테스트 케이스·통합 테스트 결과서를 작성하고, 이 문서는 배포 직전 최종 확인용으로 활용한다.

## 작성 시점

시험 단계 — 배포 1주 전에 초안을 작성하고, 검증 수행 후 Pass/Fail 결과와 결함을 즉시 기입하여 갱신한다.

## 필수 섹션 구조

### 1. 기능 검증 항목

핵심 기능별 Pass/Fail 체크 목록을 작성한다. QA ID, 기능 구분, 검증 항목, 검증 방법, 결과(Pass/Fail), 비고를 포함한다. 배포 전 반드시 Pass여야 하는 항목은 필수(★)로 표시한다.

### 2. 주요 시나리오 (E2E)

대표 사용자 흐름 3~5개를 End-to-End 시나리오로 정의하고 수행 결과를 기록한다. 각 시나리오는 시작 조건, 수행 단계, 예상 결과, 실제 결과, 판정으로 구성한다.

### 3. 비기능 점검

응답 시간, 브라우저/디바이스 호환성, 화면 레이아웃 깨짐 여부를 점검한다. 목표 수치를 사전에 정의하고 실측값과 비교한다.

### 4. 보안 기본 점검

SQL Injection, XSS, 인증 우회, 민감 정보 노출 등 기본 보안 항목을 점검한다. 자동화 스캔 도구를 사용한 경우 도구명과 결과 요약을 함께 기재한다.

### 5. 결함 기록

검증 중 발견된 결함을 즉시 기록한다. 결함 ID, 관련 QA ID, 결함 내용, 심각도, 조치 상태, 담당자를 포함한다. 결함이 없을 경우 "발견된 결함 없음"으로 명시한다.

### 6. 배포 최종 판정

전체 검증 결과를 종합하여 배포 가능 여부를 판정한다. 필수(★) 항목 전체 Pass, Critical/High 결함 0건을 배포 최소 기준으로 설정한다.

### 변경 이력

버전별 변경 내용을 추적한다.

## 작성 팁

- **QA ID 체계를 기능 구분에 맞춰 부여한다.** `QA-{기능약어}-{순번}` 형식(예: QA-AUTH-001, QA-DOC-001)으로 ID를 부여하면 결함 기록과 배포 판정 시 추적이 쉬워진다.
- **개발자가 직접 수행하지 않는다.** 개발자가 작성하고 수행하면 자신의 코드에 대한 확증 편향이 생긴다. 기획자, 디자이너, 또는 다른 팀원이 수행하게 하고 개발자는 결함 조치에만 집중한다.
- **E2E 시나리오는 실제 사용자 흐름을 기반으로 선정한다.** "회원가입 → 로그인 → 핵심 기능 사용 → 로그아웃"처럼 사용자가 실제로 수행하는 흐름을 3~5개 선정한다. 기능 단위 검증이 아닌 사용자 관점의 흐름 검증이다.
- **보안 점검은 자동화 도구와 수동 점검을 병행한다.** OWASP ZAP, Burp Suite Community 등 무료 도구로 기본 스캔을 수행하고, 도구가 잡지 못하는 인증 우회·접근 제어 오류는 수동으로 점검한다.
- **결함 심각도를 명확히 기준을 세워 적용한다.** Critical: 서비스 전체 불가, High: 핵심 기능 불가, Medium: 일부 기능 불편, Low: 미관·표기 오류 기준으로 통일하면 배포 판정 시 이견이 줄어든다.
