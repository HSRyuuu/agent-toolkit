# 업데이트 전략 패턴 가이드

`excel-doc-updater` 스킬이 전략을 추론할 때 사용하는 분류 기준. 프로파일 출력(`references/profile_schema.md`)과 사용자 의도를 종합해 다음 셋 중 하나로 결정한다.

## row_append (행 추가형)

### 시그널
- 단일 시트에 동일 헤더 아래 행이 반복된다.
- 한 컬럼에 같은 prefix를 가진 ID가 N개 누적되어 있다 (예: `PG-AI-003-001`, `PG-AI-003-002`, ...).
- 사용자 의도가 "항목들을 추가/갱신" 같이 행 단위.

### 처리 방식
1. 대상 시트에서 prefix 매칭 행을 모두 식별.
2. 첫 매칭 행의 셀 단위 스타일을 메모리에 백업.
3. 매칭 행을 모두 삭제.
4. spec 순서대로 행을 다시 삽입하면서 백업한 스타일을 셀 단위로 복제.
5. 다른 prefix의 행(보호 행)은 위치만 자동으로 밀려 그대로 보존.

### 예시
- 프로그램목록 xlsx의 "문서명" 시트에서 PG-AI-003-XXX 행만 갱신, PG-AI-001/002/004~ 행은 보존.
- 요건정의서의 기능 요구사항 행 (FR-001, FR-002, ...).

### 주의
- 행 삭제·삽입은 다른 행의 위치를 바꾼다 → 셀 좌표 기반 참조가 있는 다른 시트가 있으면 깨질 수 있음. 프로파일 단계에서 cross-sheet 수식 사용 여부를 확인.
- `=ROW()-N` 같은 위치 의존 수식이 헤더에 있으면 그대로 복제해도 자동 보정됨.

## sheet_clone (시트 복제형)

### 시그널
- 시트 이름이 같은 prefix로 반복된다 (예: `IF-AI-003-001`, `IF-AI-003-002`, ...).
- 각 시트의 양식이 동일하고, 항목별로 시트 한 장이 할당된다.
- 사용자 의도가 "항목별 시트 만들기" 또는 "양식별로 다시 만들기".

### 처리 방식
1. 마스터 시트 1개를 정한다 (보통 prefix의 첫 번째, 예: `*-001`).
2. 보호 시트 화이트리스트를 정한다 (표지, 개정이력, 다른 prefix 시트들).
3. 보호 시트가 아니고 마스터도 아닌 동일 prefix 시트를 모두 삭제.
4. spec 순서대로 마스터를 `wb.copy_worksheet()`로 복제, 시트명을 ID로 변경, 셀에 spec 데이터 채움.
5. 마스터 자신도 첫 번째 spec으로 채움.
6. 시트 순서를 원본 패턴대로 정렬 (보호 시트 → 새 시트들 → 나머지 보호 시트).

### 예시
- 인터페이스설계서 xlsx의 IF-AI-003-XXX 시트.
- 각 인터페이스/엔드포인트별로 명세 시트를 따로 두는 양식.

### 주의
- 시트 복제는 openpyxl의 `wb.copy_worksheet()`만 안전하다. 새 시트를 만들고 셀 스타일을 직접 베끼는 방식은 병합/열 너비/이미지/조건부서식이 누락된다.
- 동일 prefix 시트 수가 많으면(60개+) 메모리·실행 시간 주의.

## cell_update (셀 갱신형)

### 시그널
- 새 행/시트를 만들 필요가 없고, 정해진 셀 몇 개의 값만 바꾼다.
- 보통 헤더, 작성자/작성일, 버전 정보 같은 메타데이터 갱신.
- 사용자 의도가 "특정 칸만 채워줘" 또는 "헤더 정보 업데이트".

### 처리 방식
1. 갱신 대상 (시트, 셀좌표, 새 값) 리스트를 spec으로 받는다.
2. 각 셀에 대해 값만 갱신, 스타일은 건드리지 않는다.
3. 다른 셀은 읽지도 쓰지도 않는다.

### 예시
- 표지 시트의 "작성일" / "버전" 셀 갱신.
- 개정이력 시트의 마지막 행에 한 줄 추가 (이건 row_append로도 풀 수 있음 — 헤더 상태에 따라 결정).

### 주의
- 가장 단순하지만, "어느 셀이 정말 메타데이터인지" 추론이 어려움. 사용자에게 셀 좌표 + 새 값 매핑을 명시적으로 받는 편이 안전.

## 복합 전략

위 셋이 한 번에 필요한 경우 (예: 프로그램목록처럼 행도 추가하고 표지의 버전 셀도 갱신해야 할 때), spec을 두 종류로 분리해 두 단계로 실행한다. 한 스크립트에 섞지 말고 step을 명시적으로 나눠 사용자가 각 단계를 검토할 수 있게 한다.

## 추론이 모호할 때

다음 중 하나라도 해당되면 사용자에게 묻고 진행한다.

- prefix 매칭 행이 0개거나 너무 많아 마스터를 정하기 모호함.
- 같은 prefix가 여러 시트에 흩어져 있어 row_append인지 sheet_clone인지 불분명.
- 보호 영역 후보가 명확하지 않음 (예: 시트 이름이 모두 비슷함).
- 사용자 의도와 프로파일이 충돌함 (예: "행 추가"라고 했는데 시트 prefix가 반복됨).
