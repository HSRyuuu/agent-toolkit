# Resolve Conflict — 충돌 분류·해결 (공유 프롬프트)

> **pull**(`pull.md`), **PR 생성 전 베이스 머지**(`pr.md`), **rebase**(`rebase.md`)가 공통으로 참조하는 충돌 해결 프롬프트.

핵심은 "AI가 자신 있게 해결할 수 있는 충돌"과 "사람의 판단이 필요한 충돌"을 **엄격하게 가르는 것**이다. 잘못된 자동 해결은 조용한 데이터 손실로 이어지므로, 의심스러우면 해결하지 말고 물어본다.

## 핵심 원칙

- **충돌 해결은 보수적으로** — 양쪽 의도가 명백히 보존 가능한 자명한 충돌만 자동 해결한다. 의미가 겹치거나 의도가 불확실하면 **반드시 사용자에게 묻는다**. "엄격하게"란 확신이 없으면 자동 해결하지 않는다는 뜻이다.
- **되돌릴 수 있게** — 막히면 `git merge --abort`(또는 `git rebase --abort`)로 원상복구할 수 있음을 늘 염두에 둔다.

## 1. 충돌 분류 — 엄격한 판단

충돌 파일을 모은다:
```bash
git diff --name-only --diff-filter=U   # 충돌(unmerged) 파일 목록
git status                             # both modified / deleted by us·them 등 상태
```

각 충돌을 하나씩 **읽고**(`git show :1:<file>` 공통 조상, `:2:` 현재(ours), `:3:` 대상(theirs)을 비교) 아래 기준으로 분류한다. 각 충돌 헝크에 대해 "양쪽이 무엇을 왜 바꿨는지"를 먼저 이해한 다음 판단한다.

### 자동 해결해도 되는 충돌 (SAFE — 자신 있을 때만)

확신이 설 때만 직접 해결한다:

- **양쪽 추가가 독립적** — import 목록, 의존성 목록, 서로 다른 함수·필드·라우트 추가처럼 둘 다 보존하면 의도가 온전한 경우.
- **한쪽은 포맷/공백/주석만** — 한쪽이 실질 변경이고 다른 쪽은 들여쓰기·줄바꿈·주석뿐이면 실질 변경을 채택.
- **lockfile / 생성물** — `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `uv.lock`, `poetry.lock`, 빌드 산출물 등은 손으로 합치지 말고 **재생성**으로 해결한다(해당 패키지 매니저 명령). 재생성이 불가하면 사용자에게.
- **명백한 중복** — 양쪽이 사실상 동일한 변경을 해 한쪽만 남기면 되는 경우.

### 사용자에게 물어야 하는 충돌 (ESCALATE — 기본값)

조금이라도 아래에 해당하면 **해결하지 말고** 모아 둔다:

- **같은 코드를 다르게 수정** — 동일 함수·조건·상수·로직을 양쪽이 다른 의도로 바꾼 경우. 한쪽을 고르면 다른 쪽 의도가 사라진다.
- **비즈니스 로직·알고리즘·계산식** 변경 충돌.
- **delete/modify** — 한쪽이 파일·블록을 삭제했고 다른 쪽이 수정한 경우(`deleted by us/them`). 삭제가 의도인지 알 수 없다.
- **의도 불명** — 커밋 메시지·코드만으로 어느 쪽이 맞는지 판단이 안 서는 경우.
- **설정·스키마·마이그레이션·보안** 관련처럼 틀리면 영향이 큰 파일.
- 자동 해결이 양쪽 의도 중 하나라도 **손실**시킬 가능성이 있는 모든 경우.

> 판단 규칙: SAFE인지 1초라도 망설여지면 ESCALATE다. 자동 해결의 잘못은 눈에 안 띄고 나중에 터진다.

## 2. 해결 실행

**SAFE 충돌**: 파일을 직접 편집해 충돌 마커(`<<<<<<<`, `=======`, `>>>>>>>`)를 제거하고 의도를 보존한 결과로 만든 뒤 `git add <file>`.

**ESCALATE 충돌**: 한꺼번에 모아 사용자에게 묻는다(`AskUserQuestion`). 각 충돌에 대해 양쪽 내용을 짧게 보여주고 선택지를 제시한다 — 예: "현재(ours) 채택 / 대상(theirs) 채택 / 둘 다 / 직접 설명". 사용자 결정대로 적용하고 `git add`.

모든 충돌을 처리했으면 병합을 마무리한다:
```bash
git diff --check                 # 남은 충돌 마커 점검
git commit --no-edit             # 병합 커밋 완료 (rebase면 git rebase --continue)
```

> 사용자가 어떤 충돌도 해결하길 원치 않으면 `git merge --abort`(또는 `git rebase --abort`)로 깨끗이 되돌리고 그 사실을 보고한다.

## 안전 규칙

- 모호한 충돌을 추측으로 한쪽만 채택해 해결하지 않는다 — 반드시 사용자에게 묻는다.
- 충돌 마커가 남은 채 커밋·푸시하지 않는다 (`git diff --check`로 확인).
- 막히거나 예상과 다르면 밀어붙이지 말고 `--abort`로 되돌린 뒤 상황을 설명한다.
