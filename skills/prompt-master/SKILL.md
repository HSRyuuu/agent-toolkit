# Prompt Master

사용자의 요구사항을 분석하여 프로덕션 수준의 LLM 프롬프트를 설계한다. 프롬프트 설계는 **전략 선택**(어떤 기법을 쓸 것인가)과 **작성 기법**(어떻게 잘 쓸 것인가) 두 축으로 이루어진다.

## 작업 흐름

### 신규 프롬프트 작성

1. **목적 파악**: 프롬프트의 목적, 대상 LLM, 사용 환경(API/챗/에이전트)을 파악한다
2. **전략 선택**: 아래 "전략 선택 가이드"로 적용할 기법을 결정한다 → 상세가 필요하면 `references/prompt-techniques.md`
3. **프롬프트 설계**: "작성 기법 매트릭스"에서 적용할 기법을 확인한다 → 상세가 필요하면 `references/writing-patterns.md`
4. **자가 리뷰**: `references/review-checklist.md`로 품질을 점검한다
5. **제시 및 개선**: 완성된 프롬프트를 사용자에게 제시하고, 피드백을 반영한다

### 기존 프롬프트 리뷰/개선

1. **현행 분석**: 프롬프트의 구조, 지시사항, 예시, 출력 형식을 분석한다
2. **체크리스트 적용**: `references/review-checklist.md`로 문제점을 식별한다
3. **개선안 제시**: 문제점별로 구체적인 개선 방안과 근거를 제시한다

### 프롬프트 평가

사용자가 프롬프트 비교, 평가, 테스트를 요청할 때 `references/prompt-evals.md`를 읽고 해당 방법론을 적용한다.

## 전략 선택 가이드

프롬프트를 설계하기 전에, 작업의 성격에 맞는 프롬프팅 전략을 먼저 선택한다. 각 기법의 상세 설명과 예시는 `references/prompt-techniques.md`를 참조한다.

### 전략 선택 의사결정 트리

```
작업이 단순한가? (분류, 번역, 요약 등)
├─ YES → 모델이 잘 아는 작업인가?
│        ├─ YES → Zero-Shot (1)
│        └─ NO → Few-Shot (2)
└─ NO → 추론이 필요한가?
         ├─ YES → 추론 유형은?
         │        ├─ 단일 경로 추론 → Chain-of-Thought (3)
         │        ├─ 높은 정확도 필요 → Self-Consistency (4)
         │        ├─ 다중 경로 탐색 → Tree of Thoughts (7)
         │        └─ 수학/계산 → Program-Aided LM (13)
         └─ NO → 시스템/파이프라인 설계인가?
                  ├─ 외부 지식 필요 → RAG (8)
                  ├─ 도구 사용 필요 → ReAct (14) / ART (9)
                  ├─ 멀티스텝 작업 → Prompt Chaining (6)
                  └─ 반복 개선 필요 → Reflexion (15)
```

### 사용 환경별 기법 매핑

| 사용 환경 | 주요 기법 | 보조 기법 |
|----------|----------|----------|
| **1회성 챗 질의** | Zero-Shot, Few-Shot | CoT, Generated Knowledge |
| **API 파이프라인** | Prompt Chaining, RAG | Few-Shot, 출력 형식 강제 |
| **AI 에이전트** | ReAct, Reflexion, ART | CoT, Self-Consistency |
| **복잡한 추론** | CoT, ToT, Self-Consistency | PAL, Meta Prompting |
| **프롬프트 자동 최적화** | APE, Active-Prompt | Meta Prompting |

> Directional Stimulus(12), Multimodal CoT(16), Graph Prompting(17)은 특수 도메인(멀티모델 파이프라인, 멀티모달, 그래프 데이터) 전용 기법이다. 상세는 `references/prompt-techniques.md` 참조.

### 전략별 추천 하이퍼파라미터

| 작업 유형 | Temperature | Top-P | 근거 |
|----------|:-----------:|:-----:|------|
| 데이터 추출, JSON 출력 강제, 분류 | 0.0 ~ 0.2 | 0.9 | 결정론적 출력이 필요. 변동성을 최소화하여 파싱 안정성 확보 |
| 코드 생성, 구조화된 분석 | 0.2 ~ 0.4 | 0.9 | 약간의 유연성을 허용하되 논리적 일관성 유지 |
| 일반 챗, 문서 요약, 번역 | 0.4 ~ 0.7 | 0.95 | 자연스러운 표현을 위한 적절한 다양성 |
| 브레인스토밍, ToT, 창의적 글쓰기 | 0.8 ~ 1.0 | 1.0 | 다양한 추론 경로 탐색. Self-Consistency에서도 높은 temperature로 샘플링 |

**원칙:**
- API 파이프라인에서는 Temperature 0.0~0.2를 기본으로 설정하고, 필요 시에만 올린다
- Self-Consistency 기법은 의도적으로 temperature를 높여(0.7~1.0) 다양한 경로를 샘플링한다
- `max_tokens`는 예상 출력 길이의 1.5배로 설정하여 잘림을 방지한다

## 공통 원칙

프롬프트를 설계할 때 아래 7가지 원칙을 항상 적용한다:

1. **추상성 제거**: "복잡하면" → "3개 이상 파일이면". 모든 조건과 임계값을 수치로 명시
2. **반복 & 변주**: 같은 지시를 XML, 체크리스트, 예시 등 형태를 바꿔가며 여러 번 노출
3. **구조적 경계**: XML 태그, 역할 경계 테이블, Phase 분리로 명확한 구조 설정
4. **증거 기반**: 모든 주장에 실행 출력 등 객관적 증거 요구, 검증 없는 완료 선언 불가
5. **실패 대비 설계**: 서킷 브레이커, 상위 위임, 롤백 + 기록의 복구 프로토콜
6. **계층적 사고**: 7단계 강조 계층, 컨텍스트 배치 전략, 점진적 정보 공개
7. **자기 보고 불신**: 에이전트와 서브에이전트의 완료 주장은 관찰 가능한 증거와 독립 검증으로만 사실로 승격한다. 하네스 차원의 계약은 `agent-harness-construction`을 적용한다
