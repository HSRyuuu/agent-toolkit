# 프롬프트 평가 (Evals)

"프롬프트 A가 프롬프트 B보다 낫다"는 것을 체계적으로 증명하는 방법론이다. SKILL.md Part D의 자가 리뷰가 정성적 점검이라면, 여기서는 정량적/시스템적 평가 방법을 다룬다. 사용자가 프롬프트 평가, 비교, 테스트를 요청할 때 이 문서를 참조한다.

---

## 1. 평가 방법론

| 방법 | 설명 | 적합한 상황 |
|------|------|-----------|
| **골든 데이터셋** | 정답이 있는 테스트 케이스 N개를 만들고, 프롬프트별 정확도를 비교 | 분류, 추출, QA 등 정답이 명확한 작업 |
| **LLM-as-a-Judge** | 별도의 LLM(또는 동일 모델)에게 두 출력을 블라인드로 제시하고 어느 쪽이 더 나은지 판단시킴 | 요약, 글쓰기 등 정답이 없는 창의적 작업 |
| **A/B 테스트** | 실제 사용자 트래픽을 두 프롬프트에 분배하고 성과 지표(클릭률, 만족도 등)를 비교 | 프로덕션 환경에서의 최종 검증 |
| **자동화된 어설션** | 출력에 대한 프로그래밍적 검증 (JSON 파싱 성공률, 필수 필드 포함 여부, 길이 제한 준수 등) | API 파이프라인, 구조화된 출력 |

---

## 2. 평가 프로세스

```
1. 테스트 케이스 작성 (최소 10~20개, 엣지 케이스 포함)
2. 기존 프롬프트(Baseline)로 전체 테스트 실행
3. 개선 프롬프트(Candidate)로 동일 테스트 실행
4. 정량 비교: 정확도, 포맷 준수율, 평균 토큰 수, 레이턴시
5. 정성 비교: LLM-as-a-Judge 또는 사람 리뷰 (최소 실패 케이스 전수 확인)
6. 회귀 확인: Candidate가 Baseline에서 통과하던 케이스를 실패시키지 않는가
```

---

## 3. LLM-as-a-Judge 프롬프트 템플릿

```
아래 두 응답을 비교하여 어느 쪽이 더 우수한지 판단하세요.
평가 기준: {{CRITERIA}}

[응답 A]
{{RESPONSE_A}}

[응답 B]
{{RESPONSE_B}}

판단:
- 승자: A 또는 B
- 근거: (구체적 이유를 2-3문장으로)
- 각 항목 점수: {{CRITERIA}} 기준 1-5점
```

**주의사항:**
- 위치 편향(Position Bias) 방지를 위해 A/B 순서를 랜덤으로 섞어 2회 평가한다
- Judge 모델은 평가 대상보다 같거나 상위 모델을 사용한다
- 평가 기준(Criteria)을 사전에 구체적으로 정의해야 일관된 결과를 얻을 수 있다

---

## 4. 평가 기준 설계

평가 기준이 모호하면 Judge의 판단도 일관되지 않는다. 작업 유형별 추천 기준:

| 작업 유형 | 추천 평가 기준 |
|----------|-------------|
| 분류/추출 | 정확도(Accuracy), 필드 완전성(Completeness), 포맷 준수율 |
| 요약 | 핵심 정보 포함도, 간결성, 사실 정확성 |
| 코드 생성 | 실행 가능성, 정확성, 코드 품질(가독성/효율성) |
| 창의적 글쓰기 | 일관성, 톤 적합도, 창의성, 구조적 완결성 |
| 에이전트 행동 | 도구 사용 정확성, 단계 효율성, 최종 결과 정확도 |

---

## 5. 골든 데이터셋 작성 가이드

좋은 테스트 케이스는 프롬프트의 약점을 드러낸다. 아래 분포를 권장한다:

| 케이스 유형 | 비율 | 설명 |
|-----------|:----:|------|
| Happy Path | 40% | 일반적인 정상 입력 |
| Edge Case | 30% | 경계값, 극단적 길이, 특수문자 등 |
| Adversarial | 20% | 모호한 입력, 지시 무시 시도, 다의적 표현 |
| Regression | 10% | 이전 버전에서 실패했던 케이스 |

**테스트 케이스 형식:**

```json
{
  "id": 1,
  "input": "사용자 입력 또는 시나리오 설명",
  "expected_output": "기대하는 출력 또는 출력 조건",
  "criteria": ["정확도", "포맷 준수"],
  "category": "edge_case"
}
```
