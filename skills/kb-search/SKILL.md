# kb-search — KB 검색·질의응답

Knowledge Base의 기존 문서를 컨텍스트로 자연어 질문에 답한다.

> **Obsidian 스킬 활용**
> - `obsidian-cli` — 볼트 검색, tags 집계, 위키링크 탐색 시 활용.

---

## 핵심 원칙

1. **읽기 전용** — 파일 생성·수정·삭제·git 호출 없음.
2. **출처 명시** — 답변의 모든 사실 주장은 본문 또는 말미에 `[[경로/문서명]]`으로 인용.
3. **모른다고 말하기** — 볼트에 정보가 없으면 LLM 일반 지식으로 채우지 말고 "KB에는 정보 없음" 안내.

---

## Step 0. 볼트 경로 확인

`~/.config/kb/path`를 읽어 볼트 절대 경로를 얻는다.

```bash
test -f ~/.config/kb/path && cat ~/.config/kb/path
```

### 파일이 없거나 비어있는 경우

다음 안내 메시지를 출력하고 **사용자 응답을 기다린다** (자동 진행 금지):

```
⚠️  KB 볼트 경로가 설정되어 있지 않습니다.

검색할 대상이 없습니다. 먼저 볼트 경로를 설정하세요.

볼트로 사용할 디렉토리의 **절대 경로**를 알려주세요.
  예: /Users/you/workspace/KnowledgeBase

조건:
  - 디렉토리가 미리 존재해야 합니다.
  - 응답하면 ~/.config/kb/path 에 저장되어 모든 kb-* 스킬에서 공유됩니다.

직접 설정하려면 (대신 셸에서):
  mkdir -p ~/.config/kb && echo "/your/vault/path" > ~/.config/kb/path

설정을 원치 않으면 "취소"라고 답하세요.
```

### 사용자 응답 처리

| 응답 | 동작 |
|---|---|
| 절대 경로 (`/`로 시작) | 디렉토리 존재 확인 → 존재하면 저장 후 진행, 없으면 "디렉토리가 존재하지 않습니다: {경로}" 출력 후 재질문 |
| 상대 경로 또는 `~/...` | "절대 경로(`/`로 시작)를 입력해주세요" 출력 후 재질문 |
| "취소" 또는 빈 응답 | 즉시 종료 |

볼트가 새로 설정되었으면 그대로 검색을 시도하되, 결과가 없을 가능성이 큼을 사용자에게 안내한다.

---

## Step 1. 키워드 추출

질문 문장에서:

- 고유명사 (프로젝트명, 라이브러리명, 인명)
- 핵심 명사·동사
- 한/영 혼용 시 양쪽 모두 추출 (`RAG` ↔ `검색증강생성`)

---

## Step 2. 4계층 검색

### 계층 1 — tags frontmatter grep

```bash
grep -rl "tags:.*{키워드}" {kb_root} --include="*.md" \
  --exclude-dir=_raw --exclude-dir=.obsidian --exclude-dir=.claude
```

### 계층 2 — 제목 grep

```bash
grep -rli "^# .*{키워드}" {kb_root} --include="*.md" \
  --exclude-dir=_raw --exclude-dir=.obsidian --exclude-dir=.claude
```

### 계층 3 — 본문 grep

```bash
grep -rli "{키워드}" {kb_root} --include="*.md" \
  --exclude-dir=_raw --exclude-dir=.obsidian --exclude-dir=.claude
```

계층 1·2 결과를 우선하고, 계층 3은 보충용.

### 계층 4 — 위키링크 따라가기

상위 후보 문서 안의 `[[...]]` 링크를 따라가서 1-hop 추가 탐색. 깊이 2 이상은 가지 않는다 (무한 루프 방지).

---

## Step 3. 후보 선별

- 계층 1·2 매칭 가중치 높음.
- 본문 내 키워드 등장 빈도 보조 점수.
- 최종 본문을 읽을 문서는 **최대 7개**로 제한.

`_raw/`, `_inbox/` 폴더의 문서는 답변 출처로 쓰지 않는다 (정리 전 자료).

---

## Step 4. 답변 합성

1. 선정된 문서 본문을 읽는다.
2. 질문에 답하는 내용을 추출·통합한다.
3. **모든 사실 주장 옆에 인라인 인용**: `... 라고 알려져 있다 [[RAG/RAG-개념]]`.

### 출력 포맷 (필수 — 출력 순서가 핵심)

터미널 사용자는 **최신 출력이 화면 하단에 보이고 위로 스크롤해야 이전 내용을 본다**. 따라서 **핵심 답변과 참조 문서는 반드시 맨 마지막**에 둔다. 상단 영역은 짧게 유지.

```markdown
(선택, 짧게) 검색 과정·발견 맥락·부가 설명
- 어느 폴더에서 N개 후보를 찾았는지
- 교차 참조한 보조 문서
- 관련된 다른 주제 등

(선택) 부가 설명·근거 풀이
- 답변에 영향을 준 보조 사실
- 주의사항·맥락 정보

---

## 답변

{질문에 대한 핵심 답변. 짧고 명료하게. 인라인 [[위키링크]] 인용 포함.}

**참조 문서**
- [[경로/문서1]] — 핵심 근거
- [[경로/문서2]] — 보조 근거
- [[경로/문서3]] — 관련 맥락
```

### 출력 순서 규칙

| 위치 | 내용 | 의무 |
|---|---|---|
| 상단 (선택) | 검색 과정·맥락·부가 설명 | optional, 짧게 |
| 중간 | `---` 구분선 + `## 답변` 헤딩 | 필수 |
| 하단 | 핵심 답변 + **참조 문서** | **반드시 맨 마지막** |

답변 본체가 길어 보일 것 같으면 상단의 부가 설명을 더 줄여서라도 **핵심 답변과 참조 문서가 화면 하단에 보이도록** 한다. 사용자가 답을 보려고 위로 스크롤하지 않게 만드는 것이 출력 포맷의 1순위 목표.

### Step 4-1. 정보 부족 시

매칭된 문서가 없거나, 있어도 질문에 답하기에 부족하면 동일한 출력 순서를 유지:

```markdown
(선택) 검색 과정 — 어디까지 찾았는지, 가까운 후보는 어떤 것인지

---

## 답변

KB에는 이 주제에 대한 정보가 부족합니다.

**가장 가까운 문서**
- [[경로/문서A]] — 일부 관련
- [[경로/문서B]] — 일부 관련

추가 자료를 모으시면 `/kb-add`로 저장할 수 있습니다.
```

LLM 일반 지식으로 빈칸을 채우지 않는다.

---

## Step 5. 후속 제안 (선택)

해당하는 게 있으면 **상단 부가 설명 영역**에 한 줄로 안내한다 (답변·참조 문서 아래에 두지 않는다 — 그 위치는 사용자가 가장 자주 보는 자리이므로 핵심 답변 외에는 비워둔다).

- 관련 문서 간 위키링크가 부족해 보이면 → "`/kb-lint --boost-links {경로}`로 그래프 보강 가능"
- 사용자가 새 자료를 제공한 흔적이 있으면 → "`/kb-add`로 저장 가능"

---

## 하지 않는 일

- ❌ 파일 쓰기 (생성·수정·삭제·이동).
- ❌ `git` 호출.
- ❌ frontmatter 갱신.
- ❌ 일반 지식으로 답변 보충.

---

## 예시

### 짧은 사실 질의 (답변이 1줄)

사용자: `/kb-search 환경공단 접속 id pw`

```
접속정보/ 폴더에서 환경공단 관련 문서 1건 발견. 본문 표에서 prod·stg 두 행 확인.

---

## 답변

| 환경 | URL | id | pw |
|---|---|---|---|
| prod | https://example.prod | admin | **** |
| stg | https://example.stg | admin | **** |

**참조 문서**
- [[접속정보/환경공단_접속정보]] — 자격증명 표
```

### 개념 설명 질의 (답변이 단락)

사용자: `/kb-search EVAX에서 GraphIO 메시지 프로토콜이 어떻게 동작해?`

```
tags:evax + graphio 매칭 3개, 본문 grep 1개, 위키링크 따라가 1개 추가.
관련 회의록에서 결정 사유도 발견.

---

## 답변

EVAX의 GraphIO는 노드 간 메시지를 ProtocolBuf로 직렬화하여
TCP 위에서 length-prefixed 프레임으로 주고받는다 [[EVAX/graphio-spec]].
헤더에는 ... [[EVAX/graphio-header-layout]].

**참조 문서**
- [[EVAX/graphio-spec]] — 프로토콜 핵심
- [[EVAX/graphio-header-layout]] — 헤더 포맷
- [[EVAX/회의록/2026-04-회의록]] — 결정 사유
```

### 핵심 규칙

- 사용자가 터미널을 위로 스크롤하지 않고 **그대로 화면 하단**에서 답변과 참조 문서를 읽을 수 있어야 한다.
- 상단 부가 설명은 **짧고 비워둘 수 있음**. 길어지면 답변이 화면 밖으로 밀려난다.
