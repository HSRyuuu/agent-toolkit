# kb-search — KB 검색·질의응답

Knowledge Base의 기존 문서를 컨텍스트로 자연어 질문에 답한다.

> **Obsidian 스킬 활용**
> - `obsidian-cli` — 볼트 검색, tags 집계, 위키링크 탐색 시 활용.

---

## 핵심 원칙

1. **읽기 전용** — 파일 생성·수정·삭제·git 호출 없음.
2. **출처 명시** — 답변의 모든 사실 주장은 본문 또는 말미에 `[[경로/문서명]]`으로 인용.
3. **모른다고 말하기** — 볼트에 정보가 없으면 LLM 일반 지식으로 채우지 말고 "KB에는 정보 없음" 안내.

---

## Step 0. 볼트 경로 확인

`~/.config/kb/path`를 읽어 볼트 절대 경로를 얻는다.

```bash
test -f ~/.config/kb/path && cat ~/.config/kb/path
```

### 파일이 없거나 비어있는 경우

다음 안내 메시지를 출력하고 **사용자 응답을 기다린다** (자동 진행 금지):

```
⚠️  KB 볼트 경로가 설정되어 있지 않습니다.

검색할 대상이 없습니다. 먼저 볼트 경로를 설정하세요.

볼트로 사용할 디렉토리의 **절대 경로**를 알려주세요.
  예: /Users/you/workspace/KnowledgeBase

조건:
  - 디렉토리가 미리 존재해야 합니다.
  - 응답하면 ~/.config/kb/path 에 저장되어 모든 kb-* 스킬에서 공유됩니다.

직접 설정하려면 (대신 셸에서):
  mkdir -p ~/.config/kb && echo "/your/vault/path" > ~/.config/kb/path

설정을 원치 않으면 "취소"라고 답하세요.
```

### 사용자 응답 처리

| 응답 | 동작 |
|---|---|
| 절대 경로 (`/`로 시작) | 디렉토리 존재 확인 → 존재하면 저장 후 진행, 없으면 "디렉토리가 존재하지 않습니다: {경로}" 출력 후 재질문 |
| 상대 경로 또는 `~/...` | "절대 경로(`/`로 시작)를 입력해주세요" 출력 후 재질문 |
| "취소" 또는 빈 응답 | 즉시 종료 |

볼트가 새로 설정되었으면 그대로 검색을 시도하되, 결과가 없을 가능성이 큼을 사용자에게 안내한다.

---

## Step 1. 키워드 추출

질문 문장에서:

- 고유명사 (프로젝트명, 라이브러리명, 인명)
- 핵심 명사·동사
- 한/영 혼용 시 양쪽 모두 추출 (`RAG` ↔ `검색증강생성`)

---

## Step 2. 4계층 검색

### 계층 1 — tags frontmatter grep

```bash
grep -rl "tags:.*{키워드}" {kb_root} --include="*.md" \
  --exclude-dir=_raw --exclude-dir=.obsidian --exclude-dir=.claude
```

### 계층 2 — 제목 grep

```bash
grep -rli "^# .*{키워드}" {kb_root} --include="*.md" \
  --exclude-dir=_raw --exclude-dir=.obsidian --exclude-dir=.claude
```

### 계층 3 — 본문 grep

```bash
grep -rli "{키워드}" {kb_root} --include="*.md" \
  --exclude-dir=_raw --exclude-dir=.obsidian --exclude-dir=.claude
```

계층 1·2 결과를 우선하고, 계층 3은 보충용.

### 계층 4 — 위키링크 따라가기

상위 후보 문서 안의 `[[...]]` 링크를 따라가서 1-hop 추가 탐색. 깊이 2 이상은 가지 않는다 (무한 루프 방지).

---

## Step 3. 후보 선별

- 계층 1·2 매칭 가중치 높음.
- 본문 내 키워드 등장 빈도 보조 점수.
- 최종 본문을 읽을 문서는 **최대 7개**로 제한.

`_raw/`, `_inbox/` 폴더의 문서는 답변 출처로 쓰지 않는다 (정리 전 자료).

---

## Step 4. 답변 합성

1. 선정된 문서 본문을 읽는다.
2. 질문에 답하는 내용을 추출·통합한다.
3. **모든 사실 주장 옆에 인라인 인용**: `... 라고 알려져 있다 [[03_Resources/RAG/RAG-개념]]`.
4. 답변 마지막에 참조 문서 목록을 정리한다.

답변 포맷:

```markdown
{질문에 대한 합성 답변. 인라인 [[위키링크]] 인용 포함}

**참조 문서**
- [[경로/문서1]] — 핵심 근거
- [[경로/문서2]] — 보조 근거
- [[경로/문서3]] — 관련 맥락
```

### Step 4-1. 정보 부족 시

매칭된 문서가 없거나, 있어도 질문에 답하기에 부족하면:

```
KB에는 이 주제에 대한 정보가 부족합니다.

가장 가까운 문서:
- [[경로/문서A]] — 일부 관련
- [[경로/문서B]] — 일부 관련

추가 자료를 모으시면 `/kb-add`로 저장할 수 있습니다.
```

LLM 일반 지식으로 빈칸을 채우지 않는다.

---

## Step 5. 후속 제안 (선택)

답변 끝에 다음 중 해당하는 것이 있으면 한 줄로 안내:

- 관련 문서 간 위키링크가 부족해 보이면 → "`/kb-lint --boost-links {경로}`로 그래프 보강 가능"
- 사용자가 새 자료를 제공한 흔적이 있으면 → "`/kb-add`로 저장 가능"

---

## 하지 않는 일

- ❌ 파일 쓰기 (생성·수정·삭제·이동).
- ❌ `git` 호출.
- ❌ frontmatter 갱신.
- ❌ 일반 지식으로 답변 보충.

---

```
→ 본문 grep으로 1개 추가
→ 위키링크 따라가 1개 더

TCP 위에서 length-prefixed 프레임으로 주고받는다 [[01_Projects/EVAX/graphio-spec]].
헤더에는 ... [[01_Projects/EVAX/graphio-header-layout]].

**참조 문서**
- [[01_Projects/EVAX/graphio-spec]] — 프로토콜 핵심
- [[01_Projects/EVAX/graphio-header-layout]] — 헤더 포맷
- [[01_Projects/EVAX/회의록/2026-04-회의록]] — 결정 사유
```
