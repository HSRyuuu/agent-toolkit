# Test Sync Verifier

코드 변경 후 테스트를 실행하고, 실패하는 테스트를 분석하여 **테스트 코드만 수정**하는 스킬.
프로덕션 코드의 문제는 수정하지 않고 사용자에게 리포트한다.

## 핵심 원칙

1. **프로덕션 코드 기본 수정 금지** - `src/`, `lib/`, `app/` 등 프로덕션 소스는 읽기만 한다. 사용자가 명시적으로 수정을 요청한 경우에만 수정 가능
2. **테스트 코드만 수정** - `test/`, `tests/`, `__tests__/`, `spec/`, `*Test.*`, `*Spec.*`, `*.test.*`, `*.spec.*`, `*_test.go`, `*_test.rs` 파일만 수정 가능
3. **프로덕션 문제는 리포트** - 프로덕션 코드에 버그나 누락이 있으면 수정하지 말고 사용자에게 알린다

## 워크플로우

### Phase 1: 변경사항 파악

1. `git diff` 및 `git diff --cached`로 변경된 파일 목록 확인
2. 변경된 파일을 **프로덕션 코드**와 **테스트 코드**로 분류
3. 변경의 의도 파악 (새 기능, 버그 수정, 리팩토링 등)

### Phase 2: 테스트 실행

프로젝트의 빌드 도구를 자동 감지하여 테스트 실행:

| 감지 파일 | 테스트 명령 |
|-----------|------------|
| `build.gradle`, `build.gradle.kts` | `./gradlew test` |
| `pom.xml` | `./mvnw test` 또는 `mvn test` |
| `pyproject.toml` (pytest) | `pytest` |
| `setup.py`, `setup.cfg` | `python -m pytest` |
| `package.json` | `npm test` 또는 `yarn test` |
| `Cargo.toml` | `cargo test` |
| `go.mod` | `go test ./...` |

실행 결과에서 **실패한 테스트**와 **에러 메시지**를 수집한다.

### Phase 3: 실패 분석 및 분류

각 실패 테스트를 다음 카테고리로 분류:

**Category A - 테스트 수정 필요** (직접 수정)
- 프로덕션 코드의 정당한 변경으로 인해 테스트 기대값이 달라진 경우
- 메서드 시그니처 변경으로 테스트 호출부 업데이트 필요
- 새 필수 파라미터 추가로 테스트의 setup/fixture 수정 필요
- 리팩토링으로 클래스/메서드명이 변경된 경우

**Category B - 프로덕션 코드 문제** (리포트만)
- 기존 동작이 의도치 않게 깨진 경우 (regression)
- null/빈값/경계값 등 엣지케이스 미처리
- 예외 처리 누락
- 비즈니스 로직 오류

### Phase 4: 테스트 수정 (Category A만)

Category A로 분류된 테스트만 수정한다:
- 기대값(expected value)을 새 동작에 맞게 업데이트
- mock/stub 설정을 변경된 인터페이스에 맞게 조정
- 테스트 setup/teardown을 새 요구사항에 맞게 수정
- import 경로나 클래스명 변경 반영

수정 후 테스트를 다시 실행하여 통과 확인. 재실행에도 실패하면 해당 테스트를 Category B로 재분류하여 리포트한다.

### Phase 5: 리포트

다음 형식으로 결과를 보고:

```
## 변경 검증 결과

### 테스트 실행 요약
- 전체: N개 | 성공: N개 | 실패: N개 | 수정됨: N개

### 수정된 테스트 (Category A)
- `파일:라인` - 수정 내용 요약

### 프로덕션 코드 이슈 (Category B)
- [REGRESSION] `파일:라인` - 설명
- [EDGE CASE] `파일:라인` - 설명
- [BUG] `파일:라인` - 설명

### 권장 사항
- 프로덕션 코드에서 수정이 필요한 부분에 대한 구체적 제안
```

### Phase 6: 프로덕션 코드 수정 제안 (Category B가 있는 경우)

Category B 이슈가 존재하면 리포트 후 사용자에게 질문:

> "프로덕션 코드에서 다음 이슈가 발견되었습니다. 프로덕션 코드를 수정할까요?"

사용자가 수정을 요청하면 해당 프로덕션 코드를 수정하고 테스트를 다시 실행한다.
사용자가 거부하면 현재 상태로 종료한다.

## 판단 기준: 테스트 수정 vs 프로덕션 이슈 리포트

테스트 수정이 적절한 경우:
- 변경 전 코드에서 테스트가 통과했고, 의도된 변경으로 인해 기대값만 달라진 경우
- 테스트가 구현 세부사항(메서드명, 파라미터 순서 등)에 의존하고 있었던 경우

프로덕션 이슈로 리포트하는 경우:
- 변경과 무관한 기존 테스트가 실패하는 경우
- 변경의 명시된 의도와 실제 동작이 다른 경우
- 테스트가 합리적인 비즈니스 규칙을 검증하는데 실패하는 경우
