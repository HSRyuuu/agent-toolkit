# kb-lint — KB 건강 점검 + 링크 보강

볼트 전체를 스캔해 일관성·완전성 문제를 보고하고, 명시 옵션이 있을 때 위키링크 양방향 보강을 수행한다.

> **Obsidian 스킬 활용**
> - `obsidian-cli` — 볼트 검색, backlink·tags 집계, 위키링크 탐색.
> - `obsidian-markdown` — 위키링크 4종 변형(`[[x]]`·alias·anchor·embed) 문법 참조.

---

## 핵심 원칙

1. **기본은 읽기 전용 보고**. 자동 수정 없음. 쓰기는 `--boost-links` 같은 명시 옵션 + 사용자 컨펌이 있을 때만.
2. **양방향 강제 (위키링크 보강)**. A → B 추가 시 B → A도 추가. 그래프의 무방향 엣지로 작동.
3. **추가만, 수정 없음**. 위키링크 추가는 각 문서 끝의 `## 관련` 섹션에 행 추가만. 기존 본문 한 글자도 바꾸지 않는다.
4. **일괄 컨펌**. 변경 대상 목록을 한 번에 제시하고 사용자 승인 후 진행.
5. **git에 손대지 않음**.
6. **atomic write** — 모든 쓰기는 `.tmp` → `mv`.

---

## Step 0. 볼트 경로 확인

`~/.config/kb/path`를 읽어 볼트 절대 경로를 얻는다.

```bash
test -f ~/.config/kb/path && cat ~/.config/kb/path
```

### 파일이 없거나 비어있는 경우

다음 안내 메시지를 출력하고 **사용자 응답을 기다린다** (자동 진행 금지):

```
⚠️  KB 볼트 경로가 설정되어 있지 않습니다.

점검 대상이 없습니다. 먼저 볼트 경로를 설정하세요.

볼트로 사용할 디렉토리의 **절대 경로**를 알려주세요.
  예: /Users/you/workspace/KnowledgeBase

조건:
  - 디렉토리가 미리 존재해야 합니다.
  - 응답하면 ~/.config/kb/path 에 저장되어 모든 kb-* 스킬에서 공유됩니다.

직접 설정하려면 (대신 셸에서):
  mkdir -p ~/.config/kb && echo "/your/vault/path" > ~/.config/kb/path

설정을 원치 않으면 "취소"라고 답하세요.
```

응답 처리는 다른 kb 스킬과 동일 — 절대 경로 + 디렉토리 존재 확인 후 저장.

## Step 1. 모드 분기

| 입력 | 모드 |
|---|---|
| `/kb-lint` (인자 없음) | **report** — 전체 점검 보고서 (읽기 전용) |
| `/kb-lint --boost-links` | **boost** — 최근 7일 추가분에 대해 위키링크 양방향 보강 |
| `/kb-lint --boost-links {파일}` | **boost** — 특정 파일에 대해 위키링크 보강 |
| `/kb-lint --boost-links --days 30` | **boost** — 최근 30일 윈도우 |
| `/kb-lint --boost-links --folder {경로}` | **boost** — 특정 폴더 하위 전체 |

---

## Step 2. 전체 파일 스캔 (공통)

`{kb_root}` 전체에서 `.md` 파일을 수집한다. 제외: `_raw/`, `_inbox/`, `.obsidian/`, `.claude/`, `docs/`.

(주의: `_raw/` 자체의 고아 점검 시에는 `_raw/` 안 파일을 따로 수집한다 — 점검 항목 3-6 참조.)

수집 정보:
- 파일 경로
- frontmatter (`tags`, `source`, `created`)
- 본문 길이 (line count)
- 본문 내 위키링크 목록 (4종 변형 모두)
- 인바운드/아웃바운드 링크 그래프

---

## Step 3. 점검 항목 (report 모드)

### 3-1. 깨진 위키링크

`[[X]]` 형식의 모든 링크 중 X가 볼트 내 존재하지 않는 경우. 4종 변형(`[[x]]`, `[[x|alias]]`, `[[x#sec]]`, `![[x]]`) 모두 검사.

**심각도**: error

### 3-2. 고아 페이지

인바운드 위키링크 0개인 문서. 자기 자신은 제외.

**심각도**: info

### 3-3. tags 누락

frontmatter에 `tags` 필드가 없거나 빈 배열인 문서. (`_raw/` 제외)

**심각도**: error

### 3-4. 중복 의심 문서

제목 또는 본문이 80% 이상 유사한 문서 쌍. LLM 판단으로 후보 추출 후 보고서에 안내.

**심각도**: info — `/kb-add` modify 모드로 통합 검토 권장

### 3-5. `_inbox/` 방치

`_inbox/`에 파일이 1개 이상 있으면 보고.

```
_inbox/에 N개 파일이 정리 대기 중입니다.
  → /kb-add (인자 없이) 호출로 일괄 정리 가능.
```

**심각도**: info

### 3-6. `_raw/` 고아 원본

`_raw/` 안의 파일 중, **어떤 문서의 `source: "[[_raw/X]]"` frontmatter나 본문 백링크(`원본: [[_raw/X]]`)에도 참조되지 않는** 파일.

```bash
# 참조 검색 (위키링크 4종 변형 모두)
grep -rl "\[\[_raw/{파일명}" --include="*.md" {kb_root} --exclude-dir=_raw
```

미참조 _raw 파일을 보고:

```
_raw/에 고아 원본 N개:
  - {파일명} (마지막 수정: {date})
    → 정리 문서가 없거나, source 참조가 끊겼을 수 있음.
```

**심각도**: warn

### 3-7. 진부한 정보

frontmatter `created`가 **6개월 이상** 지난 문서 중, 다음 tags를 포함하는 경우:

`python`, `fastapi`, `langchain`, `langgraph`, `spring`, `sqlalchemy`, `nextjs`, `react`, `pydantic` 등 버전 의존성이 큰 기술.

내용이 최신 버전과 맞지 않을 가능성. 리뷰 권장.

**심각도**: info

### 3-8. 빈약한 문서

frontmatter를 제외한 본문이 **10줄 미만**이고 tags가 있는 문서. (단편 메모는 정상이므로 강제 권고는 아님)

**심각도**: info

### 3-9. 폴더 컨벤션 위반

- 루트의 폴더 이름이 `_`로 시작하지 않는데 비-주제 폴더로 보이는 경우 (예: `archive/`, `inbox/`, `raw/` 같은 prefix 누락)
- 루트의 폴더 이름이 `_`로 시작하는데 일반 주제 콘텐츠가 들어있는 경우 (예: `_python/`에 일반 학습 노트)

**심각도**: warn

### 3-10. 위키링크 보강 제안

각 문서에 대해 관련 후보 문서(제목·tags 키워드 매칭)가 있지만 아직 양방향 위키링크로 연결되지 않은 경우를 탐지. 보고서에 후보 쌍을 출력하고 `/kb-lint --boost-links`로 적용 가능함을 안내.

**심각도**: info

## Step 4. 보고서 출력 (report 모드)

```
===== KB Lint Report — 2026-05-22 =====

볼트: /Users/you/workspace/KnowledgeBase
총 문서 수: N (_raw 제외)

점검 항목                       | 발견 | 심각도
-------------------------------|------|------
깨진 위키링크                   | {n}  | error
고아 페이지                     | {n}  | info
tags 누락                       | {n}  | error
중복 의심                       | {n}  | info
_inbox 방치                     | {n}  | info
_raw 고아 원본                  | {n}  | warn
진부한 정보 (6개월+)            | {n}  | info
빈약한 문서                     | {n}  | info
폴더 컨벤션 위반                | {n}  | warn
위키링크 보강 제안              | {n}  | info
=============================================

다음 액션:
  /kb-lint --boost-links   ← 위키링크 자동 보강 (양방향)
  /kb-add (인자 없이)       ← _inbox 정리
  Obsidian에서 직접 수정    ← 깨진 링크·tags 누락
```

각 항목의 **상세 내역**(파일 경로·문제 위치)을 보고서 하단에 섹션별로 출력. 너무 많을 경우 상위 10개 + 전체 N개 안내.

## Step 5. `--boost-links` 모드 (구 kb-link 동작)

### 5-1. 대상 선정

| 입력 | 대상 |
|---|---|
| `--boost-links` (단독) | `created` frontmatter 최근 7일 문서 |
| `--boost-links --days N` | 최근 N일 |
| `--boost-links {파일}` | 해당 문서 1개 |
| `--boost-links --folder {경로}` | 해당 폴더 하위 전체 |

대상 0개면 종료:

> "보강할 대상이 없습니다."

### 5-2. 후보 탐색

각 대상 D에 대해:

1. D의 키워드 추출: frontmatter `tags`, 제목, `## ` 섹션 제목, 본문 고유명사.
2. 볼트에서 매칭되는 다른 문서 찾기:
   - tags 교집합 (1개 이상)
   - 제목·섹션 키워드 일치
   - 본문 grep 보조
3. **이미 연결된 경우 제외** — 위키링크 4종 변형 모두 검사:

   | 형식 | 예 |
   |---|---|
   | 일반 | `[[B-경로]]` |
   | alias | `[[B-경로\|표시텍스트]]` |
   | anchor | `[[B-경로#섹션]]` |
   | embed | `![[B-경로]]` |

   D 본문에서 위 4종 중 어느 하나라도 B를 가리키면 "이미 연결됨"으로 판정. 정규식 예: `!?\[\[B-경로(\|[^\]]*)?(#[^\]]*)?\]\]`.

4. **상위 5개**까지만 후보로 채택.

`_raw/`, `_inbox/`, `_archive/`, `docs/`, `.obsidian/`, `.claude/`는 후보에서 제외.

### 5-3. 일괄 제시 + 컨펌

```
=== KB Lint Boost Links 계획 ===

대상: RAG/rag-단점.md
  + [[RAG/RAG-개념]]            (tags 3개 일치)
  + [[RAG/벡터DB-비교]]          (제목 키워드 일치)
  + [[cache-augmented-LLM]]     (섹션 키워드 일치)

대상: cache-augmented-LLM.md
  + [[RAG/rag-단점]]            (양방향 강제)

총 영향 문서: 5개 (대상 2 + 추가될 양방향 상대 3)
진행할까요? [Y/n/부분선택]
```

- **승인** → Step 5-4
- **거부** → 종료
- **부분 선택** → `1,3` 같은 번호. 선택된 쌍에 대해서만 진행.

### 5-4. 양방향 링크 추가

각 승인된 쌍 (A, B)에 대해:

**A 파일에 B 링크 추가:**

A 본문 끝에 `## 관련` 섹션 있으면 → 그 안에 `- [[B-경로]]` 추가.

없으면 → 파일 끝에 새 섹션 추가:

```markdown

## 관련  <!-- linked 2026-05-22 -->

- [[B-경로]]
```

**B 파일에도 동일하게 `[[A]]` 추가** (양방향 강제).

### 5-5. atomic write

```bash
tmpfile=$(mktemp "{대상}.XXXXXX.tmp")
# tmpfile에 전체 새 내용 작성
mv "$tmpfile" "{대상}"
```

부분 append가 아니라 전체 파일을 새로 쓴 뒤 rename.

### 5-6. qmd 인덱스 리빌드 (best-effort)

모든 쌍 처리 후 1회만 실행. 실패 무시.

```bash
qmd index rebuild 2>/dev/null || true
```

### 5-7. 결과 요약

```
KB Lint Boost Links 완료.
- 처리한 문서: 5개
- 추가된 양방향 엣지: 3쌍 (6개 링크)
- 새로 만든 ## 관련 섹션: 2개

Obsidian 그래프 뷰에서 확인하세요.
```

## 하지 않는 일

- ❌ report 모드에서 어떤 파일도 수정하지 않음 (읽기 전용).
- ❌ 깨진 위키링크 자동 수정 — 사용자가 Obsidian에서 직접 (의도 모름).
- ❌ tags 누락 자동 채움 — 사용자 판단 영역.
- ❌ 중복 문서 자동 병합 — `/kb-add` modify 모드 또는 사용자가 수동.
- ❌ `_inbox/` 자동 정리 — `/kb-add` 인자 없이 호출로 위임.
- ❌ `_raw/` 고아 자동 삭제 — 실제로 필요한 원본일 수 있음, 사용자 판단.
- ❌ 기존 본문(`## 관련` 외) 수정 — 보강 모드에서도.
- ❌ 기존 위키링크 변경·삭제.
- ❌ frontmatter 수정.
- ❌ git 호출.

---

## 트레이드오프 메모

- **5개 상한 (보강 후보)**: 노이즈 엣지 방지. 정말 많은 관련 문서가 있다면 사용자가 `--boost-links {경로}`로 명시 호출.
- **양방향 강제**: Obsidian backlink로 단방향도 검출되지만, 그래프 뷰의 시각적 풍부함을 위해 양쪽에 명시 링크.
- **`## 관련` 섹션 컨벤션**: `/kb-add`가 신규 문서 생성 시 같은 섹션을 만든다. 호환.
- **read-only가 기본**: 자동 수정의 사이드이펙트를 피하기 위해 명시 옵션 옵트인.

---

## 예시

```
# 전체 점검 (read-only)
/kb-lint
  → 10개 점검 항목 실행
  → 보고서 출력
  → 액션 제안 (boost-links / _inbox 정리 등)

# 최근 1주 추가분 위키링크 보강
/kb-lint --boost-links
  → 최근 7일 created 문서 N개
  → 후보 5개씩 탐색, 이미 연결된 건 제외
  → 일괄 컨펌
  → 양방향 링크 추가 + qmd 리빌드

# 특정 문서 위키링크 보강
/kb-lint --boost-links RAG/rag-단점.md
  → 후보 3개 발견
  → 양방향 6개 링크 추가

# 폴더 단위 보강
/kb-lint --boost-links --folder RAG
  → RAG/ 내 모든 문서에 대해 보강
```
